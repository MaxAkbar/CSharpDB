# Summary

The `v2.6.0` release focuses on three tracks: expanded multilingual collation support, ordered SQL text indexing plus overflow hardening, and a broader maintenance and release-prep pass across Admin, diagnostics, and benchmark tooling.

On the text side, this branch grows collation support from the earlier case-insensitive work into a fuller set of explicit `BINARY`, `NOCASE`, `NOCASE_AI`, and built-in `ICU:<locale>` options that flow through SQL comparison, ordering, unique enforcement, and collection path indexes. It also adds ordered SQL text index support, collation-aware index metadata and validation, and follow-up fixes so large prefix-heavy text index buckets can spill safely through overflow pages instead of destabilizing a single index page.

On the maintenance and tooling side, this branch hardens reindex rebuilds, WAL-open cleanup, stats freshness after reopen, and live-database inspection behavior. The Admin app now handles large query result sets more safely by reusing the same paging and filter surface across table browsing, ad-hoc SQL, and the query designer, while deferring expensive welcome-page row counts until requested. The branch also adds a JSON-spec-driven benchmark data generator and focused release benchmark mode so release validation is easier to reproduce.

## Notes for Reviewers

- Review the collation and ordered-text index work together. The important question is whether the richer collation surface and ordered-text index support stay consistent across SQL comparison, planner metadata, collection indexes, and repair paths.
- The overflow-page work in the ordered-text index path is release-relevant storage hardening, not just a benchmark optimization. It exists to keep large text buckets valid under real duplicate and prefix-heavy workloads.
- The Admin result-paging work is mostly about preventing avoidable full-result scans in large local databases. It is a usability and safety improvement, not a new query feature.
- `tests/CSharpDB.DataGen` and the focused benchmark mode are release-validation infrastructure. They matter for how this branch is exercised and documented, but they are not new runtime dependencies for product consumers.
