# Summary

The `v2.3.0` release focuses on four tracks: configurable WAL durability behavior, grouped durable commit throughput, pipeline and collection reliability hardening, and documentation/admin publishing improvements.

On the storage side, this branch introduces explicit file-backed WAL durability modes (`Durable` and `Buffered`), routes WAL flushing through explicit flush policies, and adds grouped durable commit coordination so concurrent durable writers can share commit work more efficiently. The branch also moves pager commit waiting out from under the writer lock and refreshes durable-vs-buffered benchmark coverage for the current comparison set.

On the runtime side, pipeline execution and the admin pipeline editor are hardened with better invalid-package handling, CSV column previews, schema-aware destination coercion, resolved file-path behavior, richer runtime diagnostics, preserved explicit nulls during table writes, and stronger resume/checkpoint coverage.

On the collection side, binary collection storage now falls back to JSON for unsupported value types, `UInt64` binary round-tripping is preserved, and unsupported typed document shapes again route through the JSON fallback path with regression coverage.

On the docs and publishing side, this branch adds the static documentation site under `www`, introduces release/publish workflow improvements for GitHub Pages plus admin/daemon publishing, and updates the admin pipeline designer with a New Pipeline action and correct static asset serving.

## Related Issues

- File-backed WAL commits now have explicit durability modes so callers can choose `Durable` or `Buffered` behavior instead of relying on a single fixed flush policy.
- WAL flushing now goes through explicit flush-policy handling in the storage layer, making the durable-vs-buffered distinction a first-class storage-engine concern.
- Durable writes now support grouped commit coordination so multiple writers can share a single durable flush sequence instead of each forcing fully independent commit completion.
- Pager commit waiting was moved out from under the writer lock to reduce unnecessary serialization during durable commit completion.
- WAL/storage coverage was expanded around durability-mode behavior, grouped durable commit visibility, and failure behavior.
- Benchmark wiring now carries durability selection through the benchmark harness, and refreshed durable-vs-buffered benchmark reports were added for `v2.3` and the current comparison set.
- Pipeline execution now resolves file-based component paths more consistently and reports richer transform/destination failure context.
- Pipeline writes now apply schema-aware destination coercion and preserve explicit `null` values during table writes.
- The admin pipeline editor now preserves valid package state, previews CSV source columns, and avoids resetting user work when package JSON is invalid or empty.
- Pipeline coverage was expanded for relative file resolution, failure reporting, stored transform persistence, schema-driven client writes, and resume/checkpoint rewind behavior.
- Collection binary storage now falls back to JSON when a value type is not supported by the binary collection path.
- Collection codec regressions were fixed so `UInt64` values preserve round-tripping in binary collection decoding.
- JSON fallback for unsupported typed document shapes was restored in collection decoding, with regression tests covering the review findings.
- Added a static documentation site under `www` covering landing pages, architecture, benchmarks, getting-started guidance, storage, SQL, pipelines, collections, and samples.
- Added a storage internals walkthrough script in `docs` for release and educational content.
- Added a publish helper script for `CSharpDB.Admin` and `CSharpDB.Daemon`.
- Updated the release workflow to deploy the documentation site to GitHub Pages and added the supporting Pages workflow.
- Added a New Pipeline action in the admin UI pipeline designer and enabled static asset mapping so site assets are served correctly.

## Notes for Reviewers

- The main storage-level user-facing change in this branch is that file-backed WAL durability is now explicit rather than implicit: `Durable` and `Buffered` are both supported and intentionally benchmarked/documented as different operating modes.
- Group commit is specifically about durable writes. Buffered mode keeps its lighter-weight behavior and should not be read as taking the same grouped durable completion path.
- The grouped commit work is paired with the writer-lock change so the branch should be reviewed as a write-path coordination improvement, not as a standalone benchmark tweak.
- The pipeline work spans both runtime behavior and the admin editor. Review it as one reliability pass: invalid package handling, file resolution, coercion, failure context, and resume behavior are all part of the same hardening effort.
- The collection fixes near the tip of the branch are targeted regression fixes on top of the broader collection binary-storage work. They intentionally preserve JSON fallback behavior for shapes or value types that the binary path does not support cleanly.
- The docs/site changes are not only content additions. They also include the workflow and hosting changes needed to publish the static documentation site through GitHub Pages.
