## Summary

This `v3.0.0` release brings source-generated collection models into the engine,
adds a trim-safe generated collection API, carries forward the latest merged
`main` engine and CLI improvements, and includes a focused recovery of
collection write-path performance after the branch merge.

The biggest functional addition is generated collections. `CSharpDB.Generators`
now emits collection-model metadata that the engine can consume through
`GetGeneratedCollectionAsync<T>()` and `GeneratedCollection<T>`, allowing typed
collection usage without falling back to reflection-heavy runtime discovery.
This release also adds the supporting runtime model infrastructure, sample app,
trim-smoke validation, and expanded tests for generated collection behavior.

On the runtime side, the branch keeps the mainline optimizer, durable-write, and
CLI improvements from `main`, then follows with collection-specific write-path
recovery work. The collection path now avoids read-side B-tree routing-cache
population during write probes, reuses traversal scratch during insert/replace
operations, and buffers collection catalog mutation bookkeeping inside explicit
transactions so transactional write batches retain their performance gains
without losing row-count correctness.

## Notes for Reviewers

- Review this as a combined generated-collections and performance release, not
  just a docs or sample update. The key behavior changes are in the generated
  collection runtime/generator path and the collection write-path performance
  recovery.
- The highest-risk areas are `CollectionModelGenerator`, the generated-model
  registration/runtime plumbing in `CSharpDB.Engine`, and the collection write
  path through `Collection`, `Database`, and `BTree`.
- The collection-performance changes are intentionally narrow. The important
  checks are that collection writes avoid reintroducing the post-merge
  allocation/throughput regression while preserving correct transactional row
  counts and SQL visibility over `_col_*` backing tables.
- The planner and CLI sections of the release are largely inherited from the
  merged `main` branch and should be reviewed as already-shipped mainline
  behavior being carried forward into `version3.0.0`.
