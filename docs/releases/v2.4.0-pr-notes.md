# Summary

The `v2.4.0` release focuses on three tracks: ADO.NET metadata/tooling compatibility, `CSharpDB.Pipelines` release readiness, and a broad documentation/site accuracy pass.

On the ADO.NET side, this branch implements `DbConnection.GetSchema()` support for the standard metadata collections that schema-driven tools expect (`MetaDataCollections`, `Tables`, `Columns`, `Indexes`, and `Views`). It also exposes a typed `CSharpDbConnection.CreateCommand()` helper so concrete-provider callers get `CSharpDbCommand` directly and can use conveniences such as `Parameters.AddWithValue(...)` without extra casts.

On the packaging side, this branch adds a proper `CSharpDB.Pipelines` README with a complete end-to-end example and wires that package into the CI and tagged-release NuGet pack flow so it ships with the rest of the package set. Package metadata is also updated to point at `https://csharpdb.com/`.

On the docs side, this branch refreshes the package READMEs, updates the root `CSharpDB` README, expands the embedded-engine examples, adds a static full-text-search docs page, and corrects the `www` site examples and API reference so they reflect the current public API surface rather than older draft shapes.

## Related Issues

- `CSharpDB.Data` now implements `DbConnection.GetSchema()` metadata collection support for `MetaDataCollections`, `Tables`, `Columns`, `Indexes`, and `Views`.
- Schema metadata now filters out internal tables/indexes and returns user-visible catalog information in a form that ADO.NET tooling can consume.
- `CSharpDbConnection` now exposes a typed `CreateCommand()` helper so the concrete provider surface returns `CSharpDbCommand` instead of forcing consumers through the base `DbCommand` shape.
- Typed provider callers can now use `CSharpDbParameterCollection.AddWithValue(...)` directly from `connection.CreateCommand()` without an extra cast.
- `CSharpDB.Pipelines` now has a package README that documents the current package boundary, execution modes, JSON serialization flow, and a complete CSV-to-JSON example.
- CI and release workflow packing now include `src/CSharpDB.Pipelines/CSharpDB.Pipelines.csproj`, and the package README rewrite step now covers the new package.
- Package project metadata now points to `https://csharpdb.com/`.
- `CSharpDB.Engine`, `CSharpDB.Execution`, `CSharpDB.Primitives`, `CSharpDB`, and related docs now use examples that match the current engine, collection, storage, and query-result APIs.
- The `www` docs site and API reference now reflect the current public surface for collections, ADO.NET, client SDK usage, storage APIs, database modes, pipelines, and concurrent readers.
- The static docs site now has centralized page metadata rendering and code-block title handling to reduce duplicated head/markup boilerplate.
- Added a dedicated full-text-search docs page to the site and updated roadmap/docs references to reflect the shipped state more accurately.
- The retained SQL batched row transport design note is now explicitly marked as paused design context rather than shipped functionality.

## Notes for Reviewers

- The only runtime-facing code changes in this branch are on the `CSharpDB.Data` provider surface plus the CI/release workflow updates for `CSharpDB.Pipelines`. Most of the branch is docs/site correction work.
- Review the ADO.NET metadata implementation as a tooling-compatibility feature: the key behavior is that `GetSchema()` exposes user-visible tables/views/indexes/columns while hiding internal storage details.
- Review the typed `CreateCommand()` addition as a provider ergonomics fix, not as a new cross-provider ADO.NET abstraction.
- The `CSharpDB.Pipelines` changes are release-prep work: README content plus workflow inclusion, not a new runtime feature area.
- The website/docs pass intentionally removes or rewrites outdated examples that no longer match the shipped public API. The main thing to verify is accuracy against the current surface, not preservation of old snippet shapes.
- The current blocker for a fully green release branch is `CSharpDB.Tests`, which still reports 32 failures in the current tree across collection persistence/indexing, WAL/page-buffer behavior, hybrid file handling, maintenance, and planner/operator expectation coverage.
