# Summary

Release-prep for the `v2.2.0` pass focused on three tracks: a broader storage/runtime performance pass, a major collection storage/indexing pass, and client-surface consolidation for maintenance workflows.

On the storage/runtime side, this branch lands the opt-in mmap-backed main-file read path, lookup/write tuning presets, bounded WAL read caching, background sliced auto-checkpointing, and a broad SQL executor/read-path optimization pass with refreshed benchmark coverage and docs.

On the collection side, this branch moves the steady-state collection path onto the newer binary direct-payload format, adds direct binary extraction/hydration improvements, and broadens collection indexing/queryability with nested path indexes, multi-value array indexes, nested array-object indexes, public `FindByPathAsync(...)` / `FindByPathRangeAsync(...)`, and ordered text collection indexes for index-backed text ranges.

On the surface area side, backup and restore now ship as first-class `ICSharpDbClient` operations across direct, HTTP, gRPC, CLI, and Admin flows, and the storage inspector no longer reports a false-positive catalog decode warning on brand-new empty databases.

## Notes for Reviewers

- The branch is wider than the final backup/restore commit. Reviewers should look at it as a combined storage/runtime and surface-area release rather than as a single maintenance feature.
- The mmap work is intentionally scoped to clean main-file reads. WAL-backed pages still go through the WAL/cache path, which is why the branch also adds a dedicated WAL read cache and keeps the feature opt-in through storage options/presets.
- The SQL performance work is mostly about avoiding unnecessary row materialization and exploiting existing index/table payload structure more effectively. It is not a storage-format rewrite.
- The batched row transport document is forward-looking design work. The shipped executor changes add local batching and compact paths in targeted operators, but the public/internal operator contract is not yet fully replaced by a true batch transport.
- The collection work in this branch is not just a micro-optimization pass. It materially broadens the public collection path/query surface and moves the direct collection storage path onto the newer binary payload format.
- There is no first-class collection migrator in this release note set yet. If a user wants to rewrite existing collection data onto the current binary/path-index format immediately, the temporary recommendation is to export documents through `ScanAsync()`, serialize them to JSON, recreate indexes, and reimport through `PutAsync(...)`.
- The main maintenance-surface architectural shift is that backup/restore now belongs to the public client contract, not to host-specific direct-mode code.
- For HTTP and gRPC connections, backup and restore file paths are resolved on the remote host, not on the local CLI/Admin machine.
- Restore remains an offline replace workflow built on `LoadIntoMemoryAsync(...)` plus `SaveToFileAsync(...)`, then a file swap with rollback backup protection.
- The direct client implementation guards restore behind exclusive access by releasing the cached database handle and rejecting active snapshot readers or client-managed transactions.
- The storage-inspector fix is intentionally narrow: it only teaches diagnostics to skip the stats catalog sentinels that are not table-schema payloads.

## Temporary Collection Rewrite Guidance

For now, there is no dedicated collection migrator in this release-prep branch. If you want to rewrite existing collection data onto the current collection payload/index path using only public APIs, the recommended approach is:

1. Open the existing database.
2. Enumerate the full collection with `ScanAsync()`.
3. Save each `(key, document)` pair as JSON.
4. Open the target database.
5. Recreate the needed indexes with `EnsureIndexAsync(...)`.
6. Reimport each saved document with `PutAsync(...)`.

Public collection APIs involved:

- `Database.GetCollectionAsync<T>(...)`
- `Collection<T>.ScanAsync(...)`
- `Collection<T>.PutAsync(...)`
- `Collection<T>.EnsureIndexAsync(...)`
- `Collection<T>.FindByPathAsync(...)`
- `Collection<T>.FindByPathRangeAsync(...)`

Example JSON-lines export/import using only the public collection API:

```csharp
using System.Text;
using System.Text.Json;
using CSharpDB.Engine;

public sealed record CollectionExportRow<T>(string Key, T Document);

public static async Task ExportCollectionAsync<T>(
    Database db,
    string collectionName,
    string outputPath,
    CancellationToken ct = default)
{
    var collection = await db.GetCollectionAsync<T>(collectionName, ct);

    await using var stream = File.Create(outputPath);
    await using var writer = new StreamWriter(stream, new UTF8Encoding(false));

    await foreach (var row in collection.ScanAsync(ct))
    {
        string json = JsonSerializer.Serialize(new CollectionExportRow<T>(row.Key, row.Value));
        await writer.WriteLineAsync(json);
    }
}

public static async Task ImportCollectionAsync<T>(
    Database db,
    string collectionName,
    string inputPath,
    CancellationToken ct = default)
{
    var collection = await db.GetCollectionAsync<T>(collectionName, ct);

    using var stream = File.OpenRead(inputPath);
    using var reader = new StreamReader(stream, Encoding.UTF8);

    while (await reader.ReadLineAsync() is { } line)
    {
        var row = JsonSerializer.Deserialize<CollectionExportRow<T>>(line)
            ?? throw new InvalidOperationException("Collection export row could not be deserialized.");

        await collection.PutAsync(row.Key, row.Document, ct);
    }
}
```

Example index recreation after import:

```csharp
var users = await db.GetCollectionAsync<User>("users", ct);

await users.EnsureIndexAsync("Email", ct);
await users.EnsureIndexAsync("$.address.city", ct);
await users.EnsureIndexAsync("$.tags[]", ct);
await users.EnsureIndexAsync("$.orders[].sku", ct);
```
