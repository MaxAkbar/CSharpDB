# Summary

Release-prep for the `v2.2.0` pass focused on two tracks: a broader storage/runtime performance pass and client-surface consolidation for maintenance workflows.

On the storage/runtime side, this branch lands the opt-in mmap-backed main-file read path, lookup/write tuning presets, bounded WAL read caching, background sliced auto-checkpointing, and a broad SQL executor/read-path optimization pass with refreshed benchmark coverage and docs.

On the surface area side, backup and restore now ship as first-class `ICSharpDbClient` operations across direct, HTTP, gRPC, CLI, and Admin flows, and the storage inspector no longer reports a false-positive catalog decode warning on brand-new empty databases.

## Related Issues

- Storage read-path work now includes opt-in `UseMemoryMappedReads`, mapped clean main-file page reads with copy-on-write materialization for mutable access, mapping refresh after checkpoint growth, and a bounded WAL read cache for hot WAL-backed pages.
- File-backed tuning presets are now first-class configuration guidance: `UseLookupOptimizedPreset()` enables the current read-heavy mix, and `UseWriteOptimizedPreset()` raises the checkpoint threshold and moves checkpoint work into background slices instead of the triggering commit.
- SQL/read-path performance work in this branch includes scan filter/projection fusion, compact scan and indexed-range projection paths, batched generic expression projection, broader join lookup and covered join paths, grouped/composite index aggregate fast paths, correlated `EXISTS` / `IN` / `NOT IN` filter optimizations, filtered scalar aggregate fast paths, and lower insert metadata/front-door overhead.
- Benchmark coverage and docs were expanded around the performance pass, including refreshed `MemoryMappedReadBenchmarks`, `WalReadCacheBenchmarks`, `BTreeCursorBenchmarks`, `OrderByIndexBenchmarks`, `ScanProjectionBenchmarks`, `JoinBenchmarks`, `CompositeGroupedIndexBenchmarks`, `ColdLookupBenchmarks`, and benchmark README snapshots from March 14, 2026.
- Planning/design docs were added for the next SQL performance tier, including the SQL performance plan, shared performance phasing, and the batched row transport design. The batched transport doc is design work for the next phase, not a claim that full internal batch transport already shipped.
- `ICSharpDbClient` now exposes `BackupAsync(...)` and `RestoreAsync(...)` with shared request/result models.
- `EngineTransportClient` owns the direct backup/restore implementation and enforces exclusive-access behavior for restore.
- `CSharpDB.Api` exposes `/api/maintenance/backup` and `/api/maintenance/restore`.
- `HttpTransportClient` and `GrpcTransportClient` now cover the backup/restore surface.
- `CSharpDB.Daemon` gRPC contract and service now expose explicit backup/restore RPCs.
- CLI `.backup` / `.restore` now route through `ICSharpDbClient` instead of calling engine helpers directly.
- Admin Storage tab now routes backup/restore through `ICSharpDbClient` as well, and documents that remote paths resolve on the connected host.
- `DatabaseBackupCoordinator` remains the shared top-layer implementation detail under the direct client path.
- Storage diagnostics now skip table-stats and column-stats catalog sentinels so empty databases no longer report `CATALOG_TABLE_SCHEMA_DECODE_FAILED`.
- Backup/export/import docs, CLI docs, roadmap notes, benchmark docs, and storage-inspector docs were updated to reflect the shipped state of the branch.

## Notes for Reviewers

- The branch is wider than the final backup/restore commit. Reviewers should look at it as a combined storage/runtime and surface-area release rather than as a single maintenance feature.
- The mmap work is intentionally scoped to clean main-file reads. WAL-backed pages still go through the WAL/cache path, which is why the branch also adds a dedicated WAL read cache and keeps the feature opt-in through storage options/presets.
- The SQL performance work is mostly about avoiding unnecessary row materialization and exploiting existing index/table payload structure more effectively. It is not a storage-format rewrite.
- The batched row transport document is forward-looking design work. The shipped executor changes add local batching and compact paths in targeted operators, but the public/internal operator contract is not yet fully replaced by a true batch transport.
- The main maintenance-surface architectural shift is that backup/restore now belongs to the public client contract, not to host-specific direct-mode code.
- For HTTP and gRPC connections, backup and restore file paths are resolved on the remote host, not on the local CLI/Admin machine.
- Restore remains an offline replace workflow built on `LoadIntoMemoryAsync(...)` plus `SaveToFileAsync(...)`, then a file swap with rollback backup protection.
- The direct client implementation guards restore behind exclusive access by releasing the cached database handle and rejecting active snapshot readers or client-managed transactions.
- The storage-inspector fix is intentionally narrow: it only teaches diagnostics to skip the stats catalog sentinels that are not table-schema payloads.
