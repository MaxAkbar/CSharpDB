## Summary

This `v3.2.0` release centers on four themes: the first embedded-only EF Core
10 provider foundation, query/planner work needed to support that surface, a
focused scalar aggregate lookup optimization with refreshed guardrail
baselines, and new user-facing docs covering EF usage, programmatic insert
performance planning, and benchmark comparisons.

The biggest technical addition is `CSharpDB.EntityFrameworkCore`, which layers
an embedded EF Core provider over `CSharpDB.Data`. The branch adds
`UseCSharpDb(...)` configuration, design-time services, migrations/history/lock
plumbing, relational connection services, SQL generation, type mapping,
modification batching, database creation flow, docs, and a runnable
`samples/efcore-provider` sample. The supporting ADO.NET work also tightens
command preparation and parameter binding so the provider can execute the
required shapes cleanly.

The same branch also optimizes scalar aggregate lookup planning by reusing
lightweight lookup plans for simple indexed and primary-key-backed `COUNT(...)`
and `SUM(...)` cases, refreshes the focused scalar lookup baseline and perf
thresholds, removes the now-unused `FlushBufferedWritesAsync` hook from
`IWalFlushPolicy`, and stabilizes the benchmark guardrail workflow. On the
documentation side it adds the programmatic insert performance plan set, a
benchmark-driven CSharpDB-versus-SQLite guide, benchmark README updates, and a
small README badge refresh.

## Notes for Reviewers

- Review `src/CSharpDB.EntityFrameworkCore` as a v1 embedded provider
  foundation, not as a full general-purpose EF provider. The main questions are
  whether unsupported shapes fail early and clearly, and whether the supported
  runtime and migrations surface is coherent.
- The highest-risk cross-cutting changes are in `QueryPlanner`, `Parser`,
  `Tokenizer`, `QueryResult`, and the `CSharpDB.Data` command/binding path
  because they support the new EF translation shapes.
- `CSharpDbMigrationsSqlGenerator` and the history/lock/database-creator pieces
  intentionally support a constrained DDL set. Reviewer attention should focus
  on unsupported-operation guardrails and migration correctness, not on breadth.
- The scalar aggregate lookup work should be reviewed for plan shape and
  allocation behavior, not just the benchmark numbers. The checked-in April 18,
  2026 baseline refresh is deliberately same-runner and narrow.
- The benchmark guardrail, programmatic insert plan, and SQLite-comparison blog
  work are documentation and measurement updates on top of the runtime changes;
  they should not be mistaken for a standalone engine redesign.
