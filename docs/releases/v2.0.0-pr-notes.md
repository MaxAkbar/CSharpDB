# Summary

The `v2.0.0` compatibility and transport pass focuses on completing the REST-backed `Http` transport in `CSharpDB.Client`, filling the missing `CSharpDB.Api` surface needed for client parity, removing the unsupported `Tcp` transport placeholder, and removing the legacy `CSharpDB.Service` and `CSharpDB.Core` compatibility projects from the active `v2.0` repo surface.

This branch now also includes the SQL query-expansion and statistics work completed after that pass: set operations (`UNION`, `INTERSECT`, `EXCEPT`), scalar and correlated subqueries in the supported contexts, `ANALYZE`, persisted `sys.table_stats` and `sys.column_stats`, stale-aware column statistics, the first planner use of fresh column stats for non-unique lookup selection, and Admin dashboard shortcuts for statistics inspection.

## Notes for Reviewers

- `HttpTransportClient` now implements the `ICSharpDbClient` surface against `CSharpDB.Api` instead of failing with a not-implemented transport error.
- The API work in this pass fills the remaining gaps for remote collections, saved queries, procedure detail/execute flows, transaction sessions, checkpointing, maintenance, and storage inspection.
- `CSharpDbTransport` now exposes only `Direct`, `Http`, `Grpc`, and `NamedPipes`; `NamedPipes` still validates endpoint shape but remains intentionally not implemented.
- The legacy `CSharpDB.Service` and `CSharpDB.Core` compatibility projects are removed from the active `v2.0` repo surface rather than being kept around as migration shims.
- The `CSharpDB` meta package no longer pulls in `CSharpDB.Service`; `v2.0` consumers should target `CSharpDB.Client` and `CSharpDB.Primitives` directly.
- `CliShellOptions` accepts `http`, `grpc`, and named-pipe aliases, and HTTP is inferred from `http://` and `https://` endpoints when `Transport` is omitted.
- This pass does not collapse the REST and gRPC hosts together: `CSharpDB.Api` remains the REST/HTTP host, and `CSharpDB.Daemon` remains the dedicated gRPC host.
- SQL support now includes compound queries (`UNION`, `INTERSECT`, `EXCEPT`) plus scalar subqueries, `IN (SELECT ...)`, and `EXISTS (SELECT ...)`.
- Correlated subqueries are now supported in `WHERE`, non-aggregate projection expressions, and `UPDATE` / `DELETE` expressions. They are still rejected in `JOIN ON`, `GROUP BY`, `HAVING`, `ORDER BY`, and aggregate projections. `UNION ALL` is still not implemented.
- `ANALYZE` now refreshes exact persisted table and column stats, exposed through `sys.table_stats` and `sys.column_stats`.
- Column stats currently persist `distinct_count`, `non_null_count`, `min_value`, `max_value`, and `is_stale`. Writes mark existing column stats stale, and rollback/reopen plus VACUUM copy preserve catalog correctness.
- The planner now uses fresh column stats conservatively for non-unique equality lookup selection. This is intentionally narrow: broader range costing, join fanout estimation, and derived `sys.index_stats` are still future work.
- Added `samples/procurement-analytics`, a compact workbook sample for `UNION` / `INTERSECT` / `EXCEPT`, scalar subqueries, `IN (SELECT ...)`, `EXISTS (SELECT ...)`, `ANALYZE`, and `sys.table_stats` / `sys.column_stats`.
- `CSharpDB.Admin` now exposes `sys.table_stats` and `sys.column_stats` directly in the system catalog list and adds a one-click `ANALYZE` action plus direct stats shortcuts in query tabs.
- README, roadmap, release notes, and the table/index statistics design note were updated to reflect the current shipped surface and the remaining boundaries.
- Focused benchmark results on this branch stayed in the same band as the recent baselines: parser remained flat-to-better, query-plan-cache stayed near baseline, 1k-row point lookups remained better on the main hot paths, and the new subquery microbenchmarks are now recorded as a guardrail.
