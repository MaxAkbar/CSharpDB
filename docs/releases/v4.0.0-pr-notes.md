## Notes for Reviewers

The sharding work in version4.0.0 is explicit API-level routing. Applications must supply a route key, or Admin users must choose a route context. CSharpDB does not infer shard routes from SQL predicates, and this release does not add distributed joins, distributed writes, or distributed transactions.

The master database owns sharding metadata and catalog state. Application data remains in shard databases. Catalog edits and migration workflows write pending map changes, so active clients must be recreated or restarted before those changes become active.

Read-only fan-out is intended for diagnostics and operator workflows. Replica support is metadata-only in this release; automated replication, catch-up, promotion, and failover are not included.

NuGet release publishing now depends on the NuGet Trusted Publishing policy and the repository-level `NUGET_USER` secret.
