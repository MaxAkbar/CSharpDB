# Summary

The `v2.5.0` release focuses on three tracks: durable WAL write-path hardening and release validation, storage tuning and documentation for sustained file-backed writes, and a first phase of stats-driven cost-based join planning.

On the storage side, this branch refactors file-backed commit coordination so WAL work is staged and completed more cleanly, reduces unnecessary serialization around commit completion, hardens WAL recovery and repair behavior, and exposes the durable tuning knobs that were actually benchmarked in this branch. It also adds process-crash durability coverage plus macro benchmarks and perf guardrails for durable write regressions. The current tip of the branch also fixes a collection write-gate leak on cached collection opens that was able to stall follow-up collection count and browse calls through the HTTP and gRPC transport layers.

On the planner side, this branch introduces the first cost-based join and cardinality estimation phase. With `ANALYZE` statistics available, the planner can now make better non-unique lookup choices, pick join methods and hash build sides using estimates, and do limited greedy reordering of supported inner-join chains. Follow-up commits broaden selectivity hints for ranges, `IN`, disjunctions, nullable predicates, and mixed `UNION` cases.

On the docs and release-prep side, the branch refreshes benchmark and thread-safety guidance in the README set, documents the current storage tuning recommendations and optimizer scope, and checks in the focused durability and micro baseline material needed for reproducible release validation on compatible machines.

## Related Issues

- File-backed WAL commit coordination was refactored so commit preparation and completion are split more cleanly, durable completion work no longer holds the same lock profile as before, and recovery and repair coverage is stronger around checkpoint and WAL edge cases.
- Repeated `AppendFrameAsync(...)` calls inside a transaction are now staged and emitted as chunked WAL writes at commit time instead of taking the same append path one frame at a time.
- `StorageEngineOptionsBuilder` now exposes the durable-write tuning knobs added in this branch, including `UseDurableCommitBatchWindow(...)` and `UseWalPreallocationChunkBytes(...)`, while the storage docs guide callers toward `UseWriteOptimizedPreset()` as the current opt-in write-heavy preset.
- The storage and engine path now has more explicit runtime diagnostics around WAL flush behavior, plus new concurrency coverage for checkpoint and write interaction and reader-held WAL growth scenarios.
- Added process-crash durability harness coverage for four recovery points: immediately after commit returns, at checkpoint start, after checkpoint page flush, and after WAL finalization before pager refresh completes.
- Added durable write diagnostics and concurrent durable write benchmark suites, and expanded benchmark docs with the current tuning interpretation for batch windows, background checkpoint slices, and WAL preallocation.
- Added durability-specific perf guardrails, refreshed baseline capture and compare scripts, made guardrails machine-aware via baseline fingerprints, and checked in the focused durability and micro baseline snapshot so fresh clones can run the current guardrail set.
- Fixed a write-gate leak in `Database.GetCollectionAsync(...)` on the cached/existing collection path. Before this fix, repeated collection opens could retain the write gate and block later transport-facing collection operations like HTTP collection count and gRPC collection browse until request cancellation.
- The SQL planner now includes a first cardinality-estimation phase that uses persisted stats to guide non-unique lookup selection, join method choice, and hash build-side choice when estimates are available.
- The planner now performs limited greedy reordering of supported inner-join chains so selective predicates can be pulled earlier in the join order when stats indicate that is beneficial.
- Selectivity estimation was expanded for `BETWEEN`, `IN`, disjunctions, nullable predicate shapes, and mixed `UNION` cases, with planner regression coverage and join microbenchmarks added for those scenarios.
- The top-level, engine, execution, and storage READMEs plus the roadmap now document the current thread-safety guidance, durable and buffered performance snapshot, write-tuning recommendations, async I/O batching progress, and the current phase-1 scope of the cost-based optimizer.

## Notes for Reviewers

- Review the storage changes as one combined write-path pass: staged WAL appends, durable completion coordination, checkpoint interaction, recovery repair, and the new diagnostics and benchmarks were all changed together and are easier to assess as one unit.
- `UseDurableCommitBatchWindow(...)` and `UseWalPreallocationChunkBytes(...)` are intentionally exposed as opt-in advanced knobs, not new defaults. The docs in this branch explicitly keep both conservative by default.
- The perf-guardrail and checked-in baseline files are release-validation infrastructure. They matter for this release branch, but they are not runtime product changes.
- The cached collection-open fix near the tip is smaller than the earlier storage work but release-relevant: it resolves the transport-facing collection stalls that showed up in the API and daemon test suites after collection writes.
- The planner work is intentionally phase 1, not a full optimizer rewrite. The important review question is whether stats-driven decisions improve supported join and filter cases without regressing correctness when estimates are missing.
