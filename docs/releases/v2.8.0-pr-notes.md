# Summary

The `v2.8.0` release is centered on one main runtime track: a batch-first SQL row transport foundation across the execution engine.

This branch pushes batch-native execution deeper into the existing operator pipeline so batch-capable scans, joins, projections, filters, limits, and generic aggregate paths can keep work in row batches instead of repeatedly materializing single rows. The result is primarily an internal architecture release rather than a syntax-heavy one, but it meaningfully lowers per-row overhead, broadens reusable predicate/projection kernels, and creates the base needed for later vectorized and more specialized execution work.

Release prep also hardened this runtime path around compact-row evaluation and numeric-expression predicate binding, which fixed batch-plan regressions discovered during the validation sweep. Outside the engine, this release line adds new design docs for compiled live queries and materialized join read models, ships new site blog/news pages plus roadmap refreshes, includes a small Admin UI plumbing refactor around the title bar, modal flow, and shared database client state, and folds in the new admin-only Forms stack hosted inside `CSharpDB.Admin`.

The forms work replaces the standalone form-designer prototype with a first-class `CSharpDB.Admin.Forms` module that lives inside the main solution and is backed by the active database through the internal `__forms` metadata table. The module includes the visual designer, runtime form-entry surface, schema adaptation, validation inference, formula evaluation, JSON serialization, paged record browsing, go-to-record by primary key, column-based search, manual child-table mapping when no declared foreign key exists, and control-value coercion so checkbox/radio bindings work against text- and integer-backed boolean fields.

The release also ports and expands the dedicated forms coverage into `CSharpDB.Admin.Forms.Tests`, covering repository persistence, schema mapping, record services, serialization round-trips, generator/validation behavior, relation resolution, and admin tab integration.

## Notes for Reviewers

- Review this as an execution-engine infrastructure release first. The most important checks are that direct batch storage survives through the expected operator shapes and that compact-row evaluation stays correct when plans fuse filters and projections.
- The main user-facing additions in this range are docs/site/Admin improvements, including the integrated Forms module, not a large new SQL surface area. The runtime work is intentionally foundational.
- Release validation uncovered and fixed two real regressions in the batch path: numeric-expression predicate binding and compact-row handling in generic batch evaluation.
- This draft reflects the release range beginning at `79ef80729b009290a98a86b0f1bf2a05c6f14263` and ending at the current branch head, including the release-prep validation fixes above.
- Review this as an integration-heavy admin feature, not just a new Razor library. The important checks are admin-shell wiring, `__forms` persistence behavior, schema-signature handling, and runtime record-flow usability on nontrivial tables.
- The shipped scope is intentionally admin-only. This removes the standalone-host assumption from the prototype and keeps persistence local to the active database instead of adding public API surface.
- Child relationships now support both declared foreign keys and explicit manual mapping, but the runtime scope remains single-column parent/child mapping in this release.
- Full-solution release validation has not been rerun as part of this snapshot; the dedicated forms build and test suite were executed instead.
