## Summary

This `v3.3.0` release covers the work from
`bf954f788cc4a4c5915af2a9a68b66bc7c06f126` through
`8d11653c81e690849f462cb376872d901eabfeb6`. It focuses on durable write
performance, embedded storage tuning for ADO.NET and EF Core, same-runner
SQLite comparison coverage, and a release-stable benchmark publishing flow.

The storage work adds append-optimized row-id and hashed index maintenance
paths, hot right-edge recovery improvements, trailing-integer grouped aggregate
support, richer commit-path diagnostics, and tighter SQL index option defaults
so trailing-integer hash metadata is only applied when explicitly requested.
The branch also adds storage presets and embedded open-mode configuration for
ADO.NET and EF Core usage, with smoke and tuning tests around those public
surfaces.

The benchmark work adds SQLite C API, concurrent SQLite, concurrent ADO.NET,
strict insert, and EF Core comparison harnesses. The main benchmark README has
been redesigned around a release-core contract, generated scorecard tables, a
manifest-driven updater, and separate history/catalog/SQLite comparison files.
The current promoted benchmark set comes from the April 21, 2026 release-core
run and a clean guardrail compare: `PASS=185, WARN=0, SKIP=0, FAIL=0`.

Docs were refreshed across the root README, storage tuning guide, package-local
READMEs, SQLite comparison guide/blog, and website routing/sitemap structure.

## Notes for Reviewers

- The highest-risk runtime changes are in storage/index maintenance:
  `AppendOnlyRowIdChainStore`, hashed/appendable payload codecs,
  `OverflowingIndexStore`, `IndexMaintenanceHelper`, `QueryPlanner`, and B-tree
  insert behavior.
- `SqlIndexOptionsCodec` intentionally stops defaulting multi-column indexes to
  trailing-integer hash options. Existing serialized options are still resolved
  through the legacy compatibility path.
- SQLite and EF comparison numbers are same-runner benchmark references, not a
  universal database ranking. The authoritative published rows are in
  `tests/CSharpDB.Benchmarks/README.md` and `SQLITE_COMPARISON.md`.
- The benchmark README generated region should be edited through
  `release-core-manifest.json` plus `scripts/Update-BenchmarkReadme.ps1`, not
  manually.
- The release guardrail thresholds include narrow row-specific tolerances for
  volatile microbenchmark rows and structural `ExtraInfo` checks for commit
  fan-in behavior.
