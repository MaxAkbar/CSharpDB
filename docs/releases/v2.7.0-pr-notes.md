# Summary

The `v2.7.0` release focuses on one main runtime track: first-class foreign key constraints across the SQL engine, schema metadata, and user-facing tooling.

This branch adds v1 foreign key support for single-column, column-level `REFERENCES` definitions with optional `ON DELETE CASCADE`. Enforcement is immediate on child writes and parent deletes, and the engine now creates hidden child-side support indexes so parent checks and cascades stay bounded instead of devolving into table scans. The branch also hardens dependency handling around rename/drop paths so foreign key metadata stays consistent, and it adds `ALTER TABLE ... DROP CONSTRAINT <name>` so foreign keys are removable without a full table rebuild.

The branch also adds an older-database retrofit workflow for foreign keys. Existing databases remain readable as-is, but they do not auto-grow FK metadata on open. Instead, callers can now run a validate/apply maintenance operation that scans existing child data, reports sampled violations, optionally writes a backup, and rewrites the affected child tables with persisted FK metadata and support indexes.

On the surface area side, foreign key metadata now flows through the SQL/system-catalog path (`sys.foreign_keys`, `sys.objects`), the unified client schema models, HTTP and gRPC schema payloads, CLI `.schema`, MCP schema inspection, and ADO.NET `GetSchema("ForeignKeys")`. The docs and release notes are updated to reflect the new shipped boundary and the remaining v1 limitations.

## Related Issues

- Added parser and AST support for column-level `REFERENCES` with optional `ON DELETE CASCADE` in `CREATE TABLE` and `ALTER TABLE ... ADD COLUMN`.
- Added persisted schema metadata for foreign keys, including stable constraint identities and hidden support-index ownership.
- Added runtime enforcement for child inserts/updates, parent delete restrict behavior, and cascading parent deletes.
- Added `ALTER TABLE ... DROP CONSTRAINT <name>` support for removing foreign key constraints and their owned hidden support indexes.
- Added cycle/recursion protection for cascading deletes.
- Added `sys.foreign_keys` plus richer metadata flow through HTTP, gRPC, CLI, MCP, and ADO.NET `GetSchema("ForeignKeys")`.
- Hardened rename/drop dependency handling so referenced parents, backing indexes, and foreign-key-owned internal indexes are protected or rewritten correctly.
- Added parser, serialization, system-catalog, ADO.NET, HTTP, gRPC, CLI, Admin, and end-to-end foreign key regression coverage.
- Added `MigrateForeignKeysAsync(...)`, `POST /api/maintenance/migrate-foreign-keys`, gRPC `MigrateForeignKeys`, CLI `migrate-foreign-keys` / `.migrate-fks`, and Admin Storage-tab support for retrofitting foreign keys onto older databases.

## Notes for Reviewers

- Review this as a full-stack feature, not just a parser/runtime change. The important consistency checks are schema persistence, parent/child enforcement, hidden support-index ownership, and metadata parity across SQL, client transports, and tooling.
- The shipped foreign key scope is intentionally v1: single-column, column-level `REFERENCES` with optional `ON DELETE CASCADE`. Table-level/composite foreign keys, `ON UPDATE` actions, `SET NULL` / `SET DEFAULT`, and deferred constraints remain out of scope.
- The hidden child-side support indexes are part of the design, not an incidental implementation detail. They are what keep parent delete/restrict/cascade checks bounded for nontrivial child tables.
