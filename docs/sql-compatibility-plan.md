# SQL Compatibility Plan

Internal implementation plan for expanding the CSharpDB SQL dialect from its
current useful subset into a predictable, documented surface for real-world
applications, ADO.NET tooling, and ORMs.

This file is intentionally kept under the top-level `docs/` folder and is not
part of the public `www` documentation site. The public compatibility matrix
described below is a product deliverable and should be published under
`www/docs` as soon as its first audited baseline is ready.

## Target Outcome

CSharpDB should expose an explicit, versioned SQL dialect whose behavior can be
evaluated before an application is ported. Compatibility must mean more than
accepting syntax: supported features must execute correctly, survive reopen and
backup/restore when they affect schema, appear accurately in metadata, and work
through every applicable client surface.

The SQL compatibility roadmap is complete when CSharpDB can:

- Apply column defaults consistently across SQL, typed APIs, prepared commands,
  batches, remote clients, and the EF Core provider.
- Enforce named and unnamed `CHECK` constraints with SQL three-valued logic.
- Express and enforce table-level, composite, and action-aware foreign keys.
- Perform the common schema migrations generated by the first-party EF Core
  provider without silently weakening a model.
- Execute `UNION ALL` and a useful first set of window functions with predictable
  type, collation, ordering, and memory behavior.
- Cover the scalar and aggregate functions actually emitted by supported
  application frameworks, with documented semantics.
- Expose sufficiently complete catalogs for ADO.NET tooling, schema scripting,
  migrations, Admin, and eventual reverse engineering.
- Show the physical query plan that will run and, for read-only statements,
  report actual execution metrics through `EXPLAIN ANALYZE`.
- Publish a public, evidence-backed SQL compatibility matrix that distinguishes
  full support, partial support, planned work, and intentional exclusions.

The goal is not to claim compatibility with every vendor dialect. The goal is a
coherent CSharpDB dialect, strong compatibility with its supported providers,
and honest documentation for everything else.

## Current Baseline

Implemented behavior to preserve and extend:

- The tokenizer, parser, and AST already cover core DDL and DML, joins,
  subqueries, CTEs, aggregates, ordering, pagination, and compound queries.
- `UNION`, `INTERSECT`, and `EXCEPT` are implemented with distinct set
  semantics. `UNION ALL` is not represented in the current AST.
- Foreign keys support one child column, column-level `REFERENCES`, a primary or
  single-column unique parent key, immediate `RESTRICT`, and `ON DELETE
  CASCADE`. Rename operations repair the existing foreign-key metadata.
- `ALTER TABLE` supports add column, drop column, drop a foreign-key constraint,
  rename table, and rename column. The current drop-column path materializes and
  mutates the existing tree; it is not the shadow-root, dependency-preserving,
  crash-safe rewrite framework required by this plan.
- The scalar function evaluator has a useful set of text, null-handling,
  date/time, numeric, conversion, and Access/SQLite-style aliases. Aggregates
  include `COUNT`, `SUM`, `AVG`, `MIN`, and `MAX`.
- `ANALYZE` populates table, column, histogram, heavy-hitter, and index-prefix
  statistics used by the planner.
- `EXPLAIN ESTIMATE FOR` returns a stable diagnostic rowset containing estimate
  inputs and planner decisions without executing the target query.
- `sys.*` catalogs expose tables, columns, indexes, foreign keys, views,
  triggers, objects, temporary objects, validation rules, and planner
  statistics.
- The ADO.NET provider supports a focused `GetSchema()` surface for tables,
  columns, indexes, foreign keys, and views.
- The EF Core provider supports a deliberately narrow embedded feature set,
  including common CRUD queries, integer identity propagation, basic
  relationships, and a limited migration surface.

Known gaps and important distinctions:

- SQL `DEFAULT` column expressions and the `DEFAULT` insert forms are not yet
  modeled or executed.
- SQL `CHECK` constraints are not implemented. Existing validation rules are
  audit/reporting features; they do not reject invalid writes and must not be
  presented as check constraints.
- Table-level foreign keys, composite keys, composite foreign keys, and every
  `ON UPDATE` action are currently rejected.
- The persisted schema format and public client schema models assume a
  single-column primary key and one child/parent column per foreign key.
- Window syntax and window execution operators do not exist.
- Existing compound-query execution materializes both inputs; `UNION` then
  removes duplicates. A duplicate-preserving streaming append path does not
  exist.
- The EF Core provider currently rejects defaults, check constraints, composite
  keys and indexes, standalone foreign-key changes, `AlterColumn`, computed
  values, and many generated-SQL shapes. Unsupported schema can never be
  allowed to disappear silently from generated DDL.
- `DbDataReader.GetSchemaTable()` and ADO.NET schema collections expose too
  little provenance and constraint detail for broad tooling compatibility.
- The current explain estimator walks a diagnostic representation separate
  from the runtime operator selection. It is not a complete physical plan and
  has no actual-row instrumentation.
- Mutation paths do not expose one clearly centralized row-normalization and
  constraint-enforcement seam. Existing nullability enforcement must be audited
  while defaults and checks are routed through one shared write pipeline.
- The public roadmap describes broad feature areas, but there is no
  machine-readable, test-linked compatibility matrix at feature granularity.

## Compatibility Contract

### Non-Negotiable Rules

- Name the dialect `CSharpDB SQL` and document deviations. Do not imply blanket
  SQLite, SQL Server, PostgreSQL, ANSI, or ISO SQL compatibility.
- A parser success is not a compatibility claim. A persistent feature is only
  `supported` after execution, reopen, metadata, transport, and provider tests
  pass for the surfaces to which the claim applies.
- Unsupported syntax or model shapes must fail explicitly with an actionable
  error. They must never be ignored, approximated, or removed from generated
  DDL without the caller opting into that behavior.
- Do not add regex-based vendor-dialect rewriting or guess the intended dialect
  from SQL text. Provider-owned translation should emit CSharpDB SQL directly.
- Preserve existing valid SQL and existing on-disk databases. Schema-format
  changes must be backward-readable and covered by versioned fixtures.
- Treat newly introduced words as contextual keywords wherever practical.
  Audit existing unquoted identifiers and define the first quoted-identifier
  delimiter and escape contract in the dialect foundation before reserving a
  token globally; CSharpDB does not yet have such a contract to preserve.
- Use SQL three-valued logic, null semantics, collation, and type coercion
  consistently across filters, checks, set operations, windows, defaults, and
  foreign keys.
- Schema changes, constraint actions, and table rewrites must be atomic through
  the existing transaction and WAL boundaries. A failed or interrupted change
  must leave the original schema and rows usable.
- Row, batch, prepared, synchronous fast-path, typed CRUD, trigger, cascade,
  direct, HTTP, and gRPC paths must not acquire different constraint or
  expression semantics.
- Plan and metadata caches must be invalidated by every schema change that can
  affect binding, validation, optimization, or generated SQL.
- New compatibility paths must have bounded-memory behavior or a documented
  spill strategy before they are enabled for unbounded input.
- Every public `supported` claim must reference stable automated-test evidence.
  Partial support must name both the supported behavior and the limitation.
- Differential testing against SQLite is useful only where CSharpDB intends the
  same semantics. SQLite is a comparison target, not the definition of the
  CSharpDB dialect.

### Compatibility Status Vocabulary

The public matrix should use two independent, schema-enforced enums:

- `availability`: `supported`, `partial`, or `unsupported`.
  - `supported` means the narrow behavior is production-supported on every
    applicable listed surface and has positive automated evidence.
  - `partial` means a precisely bounded subset or only some applicable surfaces
    work. Limitations plus positive and negative/rejection evidence are
    mandatory.
  - `unsupported` means no applicable surface supports the behavior in that
    release and a clear rejection is documented.
- `roadmap`: `shipped`, `in_progress`, `planned`, `out_of_scope`, or `none`.
  This records delivery intent independently of current availability.

Each facet additionally allows `not_applicable`. The row-level availability is
derived, not hand-entered: all applicable facets supported means `supported`;
any supported/partial facet mixed with a limitation means `partial`; no
supported or partial applicable facet means `unsupported`. The JSON schema and
tests must enforce the derivation. For example, a future feature can be
`unsupported` for availability and `planned` on the roadmap.

## Primary Implementation Seams

Compatibility work crosses several existing boundaries. The implementation
should extend them deliberately instead of placing all behavior in the parser
or `QueryPlanner`:

- `src/CSharpDB.Sql`: contextual tokens, grammar, AST nodes, binding-friendly
  constraint and window representations, statement classification, and clear
  unsupported-syntax diagnostics.
- `src/CSharpDB.Primitives`: transport-neutral column, key, check, and
  foreign-key definitions; expression/function metadata; public enums.
- `src/CSharpDB.Storage`: backward-readable schema serialization, WAL-safe
  metadata updates, legacy fixtures, and crash recovery for table rewrites.
- `src/CSharpDB.Execution`: expression binding, constraint evaluation, compound
  and window operators, function semantics, physical-plan descriptors, and
  execution instrumentation.
- `src/CSharpDB.Engine`: mutation ordering, transaction atomicity, referential
  actions, dependency validation, table rewrite coordination, and cache
  invalidation.
- `src/CSharpDB.Client`, `src/CSharpDB.Api`, and protobuf contracts: additive
  schema fields and identical direct, HTTP, and gRPC behavior.
- `src/CSharpDB.Data`: ADO.NET schema collections, reader column provenance,
  generated-value handling, and provider-level compatibility tests.
- `src/CSharpDB.EntityFrameworkCore`: query translation, model validation,
  migration generation, generated-value propagation, SQL snapshots, and
  framework-version qualification.
- `src/CSharpDB.DevOps` and Admin: schema scripting/comparison, constraint
  editors, query-plan presentation, and round-trip fidelity.
- `www`, repository scripts, and CI: the public SQL reference, compatibility
  manifest, generated matrix, evidence validation, and drift checks.

## Delivery Sequence

The work should be delivered as small, independently releasable slices, but its
dependencies are important:

1. Publish the audited matrix baseline and freeze the dialect/status contract.
2. Start the independent `UNION ALL` slice while generalizing persisted schema
   metadata for defaults, checks, ordered keys, and multi-column foreign keys.
3. Implement defaults and checks, then use those foundations for referential
   actions such as `SET DEFAULT` and for richer `ALTER TABLE` operations.
4. Generalize logical candidate keys before claiming composite foreign-key
   support. The physical storage key may remain unchanged if logical composite
   primary and unique constraints are enforced correctly.
5. Build one reusable transactional table-rewrite mechanism before implementing
   several migration operations as unrelated special cases.
6. Deliver `UNION ALL` independently; it has relatively few schema
   dependencies and unlocks EF Core `Concat` patterns early.
7. Add windows in functional tiers, beginning with ranking and common aggregate
   windows, then navigation and broader frames.
8. Qualify functions and generated SQL from a captured workload corpus rather
   than an unprioritized list of function names.
9. Propagate every schema feature into catalogs and provider metadata before
   lifting provider validation errors or matrix statuses.
10. Build `EXPLAIN ANALYZE` on a shared physical-plan representation and shared
    execution instrumentation, not on the current estimate-only AST walker.
11. Gate release claims through conformance, persistence, transport, provider,
    performance, and documentation checks.

`UNION ALL` can begin immediately after the dialect baseline. After the shared
schema representation is stable, the remaining query-language track (windows
and functions) can proceed in parallel with the schema-compatibility track
(constraints, foreign keys, and `ALTER TABLE`).

## Milestones

### 1. Dialect Contract And Public Compatibility Matrix

Goal: make the current surface discoverable and testable before expanding it.

Implementation work:

- Create `www/docs/sql-compatibility.json` as the canonical machine-readable
  manifest and validate it with `www/docs/sql-compatibility.schema.json`.
- Generate `www/docs/sql-compatibility.html` from the manifest so humans and
  tools consume the same status data.
- Give every row a stable feature id and one narrow behavior. Do not use one row
  named "Foreign keys" for column-level references, composite references,
  delete actions, update actions, and deferred validation.
- Include at least:
  - category and feature name;
  - representative syntax;
  - availability and separate roadmap state;
  - `applies_to` plus explicit `not_applicable` facets;
  - parser, execution, persistence, catalog, embedded ADO.NET, REST stateless,
    REST transaction/session, gRPC unary, gRPC transaction/session, EF query,
    and EF migration facets where applicable;
  - a nullable first-supported version, which must be absent for unsupported
    rows;
  - limitations and intentional deviations;
  - documentation anchor;
  - stable positive and negative test-proof ids.
- Add top-level `verified_against` metadata containing a commit, package
  version, generation timestamp, and `development` or `released` state. Keep
  immutable released snapshots under `www/docs/sql-compatibility/releases`,
  default the public page to the latest released package, and label any
  main-branch preview unmistakably.
- Audit the current implementation and publish honest rows for the existing SQL
  subset, including the current partial foreign-key, `ALTER TABLE`, metadata,
  generated-SQL, and explain behavior.
- Freeze the CSharpDB identifier grammar, reserved/contextual keyword list,
  delimiter, escaping, case-folding, and maximum-length rules. Add positive and
  negative tests before the EF provider begins emitting quoted identifiers.
- Add `scripts/Build-SqlCompatibilityMatrix.ps1` as a deterministic generator.
  Its `-Check` mode must regenerate in memory and fail when that HTML differs
  from the checked-in HTML artifact.
- Discover evidence through a stable `SqlCompatibilityProofAttribute` or
  equivalent test trait rather than accepting arbitrary strings. Validation
  must reject missing, duplicate, orphaned, or non-CI-executed proof ids.
- Add a test that validates schema, ids, enums, derived status, required
  limitations, proof references, and the rule that a supported row has positive
  execution evidence.
- Add `scripts/Test-Documentation.ps1` to validate matrix JSON Schema, internal
  site links, navigation, and sitemap entries as one CI/Pages preflight.
- Link the matrix from the README documentation table, SQL overview, detailed
  SQL reference, docs index, roadmap, site navigation bundle, and sitemap.
- Audit and qualify blanket "Full SQL" claims across the README, public landing
  page, site bundle, docs, and blog content. Describe the supported CSharpDB SQL
  subset and link the matrix. Correct both curated and reference roadmap pages
  where a broad card overstates a narrower implemented slice.
- Update stale Admin and public help text so `ANALYZE` is described as statistics
  collection and `EXPLAIN ESTIMATE FOR` as query estimation.
- Do not publish an aggregate compatibility percentage. It conceals feature
  weighting and makes partial behavior look more complete than it is.

Acceptance criteria:

- A user can determine whether one specific syntax or behavior works without
  reading source code or a broad roadmap card.
- The latest-release view describes shipped package behavior; planned work and
  any development preview are visually and machine-readably distinct.
- `supported` rows cannot be added without at least one valid positive proof id.
- `partial` rows require positive proof and an explicit limitation or rejection
  proof.
- Persistent DDL rows require reopen and catalog evidence in addition to parser
  and execution evidence.
- The generated public page, JSON manifest, and tests cannot drift in CI.
- The release workflow snapshots the tagged manifest, verifies that its package
  version matches the release, and updates the latest-release pointer only after
  release artifacts succeed.

### 2. Persistent Constraint Model, `DEFAULT`, And `CHECK`

Goal: establish the schema representation and mutation pipeline required by
defaults, checks, generalized keys, and later migration work.

Shared schema work:

- Add persisted table, column, key, and constraint ids, then replace scalar-only
  constraint assumptions with named table constraint definitions containing
  ordered column lists.
- Add column default metadata that preserves a canonical SQL expression and the
  information needed to bind and evaluate it after reopen.
- Add named and unnamed check definitions. Normalize column checks into the
  table constraint collection while preserving their original scope for schema
  scripting.
- Add a logical ordered primary/unique key model separate from the current
  single physical `PrimaryKeyColumnIndex`. Preserve legacy single
  `INTEGER PRIMARY KEY` identity behavior, enforce `NOT NULL` on every composite
  primary-key component, and never make an integer component implicitly
  generated merely because it belongs to a composite key.
- Version `SchemaSerializer` additively and keep legacy unversioned, version 1,
  and version 2 fixtures readable. Add round-trip fixtures for new metadata and
  unknown-field handling in remote contracts.
- Define how old schemas acquire stable ids without a rewrite merely to open the
  database. Use deterministic in-memory ids based on immutable storage identity
  and ordinal/constraint position, then persist all ids atomically on the first
  schema-changing transaction before applying a rename. Detect derivation
  collisions, and preserve the persisted id across every later rename/rewrite.
- Propagate additive fields to client models, REST DTOs, protobuf messages,
  gRPC/HTTP mappers, schema signatures, archives, scripting, and comparison.
- Replace manual schema reconstruction with a canonical immutable copy/builder
  API. Audit every `TableSchema` and `ColumnDefinition` clone/rebuild site so a
  rename, temporary-table save, archive, transport mapper, or tool cannot drop
  defaults, checks, keys, ids, or referential actions.
- Store normalized expressions rather than runtime delegates. Rebind them on
  open so validation follows the same type, collation, and function rules as
  normal SQL expressions.
- Implement one row-normalization and constraint-enforcement pipeline, including
  `NOT NULL`, and route every write path through it before layering default and
  check behavior on top. Do not assume the current mutation paths already
  enforce `ColumnDefinition.Nullable` consistently.
- Define a legacy-data policy for a database that already contains nulls in a
  non-nullable column: opening for reads must not rewrite data, new/changed rows
  must be enforced immediately, validation must identify existing violations,
  and schema-altering writes to the affected table must block until the operator
  repairs or explicitly resolves them. Never coerce legacy nulls silently.

`DEFAULT` work:

- Parse column defaults in `CREATE TABLE` and `ADD COLUMN` without globally
  reserving `DEFAULT` where contextual parsing is sufficient.
- Define the first supported expression subset explicitly. It should include
  typed literals, `NULL`, and an allowlist of context-independent or
  statement-time built-ins; it should reject column references, subqueries,
  aggregates, windows, and unsafe user callbacks.
- Include `CURRENT_TIMESTAMP` in the planned statement-stable default subset and
  define its `TEXT` representation and time-zone behavior before publishing the
  example syntax.
- Evaluate defaults at insert time, not DDL time. Define whether statement-time
  values such as the current timestamp are stable for all rows in one statement
  and test that contract.
- Apply defaults only to omitted values or the explicit `DEFAULT` marker. An
  explicitly supplied `NULL` remains `NULL` and then passes or fails normal
  nullability/constraint checks.
- Add an explicit input presence contract for non-SQL APIs. Use an additive
  omitted/default/explicit-value discriminator or named-column batch API and
  carry it through client, JSON, and protobuf models; a `DbValue[]` containing
  null cannot by itself distinguish omission, `DEFAULT`, and explicit `NULL`.
- Support `INSERT ... DEFAULT VALUES` and `DEFAULT` in a values list, including
  multi-row inserts and prepared parameters around defaulted positions.
- Define precedence among explicit values, identity generation, defaults,
  nullability, checks, foreign keys, and generated-value reporting.
- Ensure typed CRUD, batch evaluation, trigger writes, import paths, EF writes,
  and remote execution use the same materialization routine.
- For `ADD COLUMN`, distinguish future-write defaults from row backfill.
  Adding a non-nullable column with a default must either rewrite every existing
  row atomically or be rejected until the rewrite path is available.
- Immediately reject unsafe `ADD COLUMN` shapes on populated tables—including a
  non-nullable, primary-key, or identity column—unless the operation uses a
  validated atomic backfill/rewrite. Do not store metadata that existing rows do
  not satisfy while broader `ALTER TABLE` work is pending.

`CHECK` work:

- Parse column-level and table-level `CHECK` clauses with optional `CONSTRAINT`
  names.
- Support deterministic, row-local Boolean expressions in the first slice.
  Reject subqueries, aggregates, windows, and functions not approved for
  constraint evaluation.
- Follow SQL three-valued semantics: `FALSE` rejects a row; `TRUE` and
  `UNKNOWN`/`NULL` pass. `NOT NULL` remains the way to reject null values.
- Bind checks against stable column ids so renames can preserve expressions and
  drops can identify dependencies safely.
- Define and document evaluation order relative to defaults, identity values,
  triggers, type conversion, nullability, and foreign keys before enabling the
  feature.
- Enforce checks on insert, update, upsert/conflict paths, prepared batches,
  typed CRUD, trigger-induced changes, cascades, table rewrites, and imports.
- Validate all existing rows before an added check becomes visible. Failure must
  report the constraint and leave the prior schema unchanged.
- Keep validation rules as a separate audit feature. They may share an
  expression evaluator, but their catalog names, enforcement, and public status
  must remain distinct.

Acceptance criteria:

- Default and check definitions survive close/reopen, backup/restore, schema
  export/import, and direct/HTTP/gRPC schema reads without semantic loss.
- Omitted, explicit `NULL`, explicit `DEFAULT`, identity, and multi-row insert
  cases produce documented and consistent values.
- A check violation fails atomically and identifies the table and constraint;
  no mutation path can bypass enforcement.
- Renaming a referenced column rewrites or rebinds constraint metadata safely;
  dropping one produces an actionable dependency error unless the constraint is
  removed in the same supported operation.
- Existing databases open without rewrite and existing valid SQL retains its
  behavior.
- Each matrix row remains narrowly scoped. A shipped literal-default row can be
  `supported` while a separate function-default row is `unsupported`; missing
  adjacent syntax must not downgrade or overstate either row.

### 3. Generalized Keys And Foreign Keys

Goal: support the key and referential-integrity shapes used by relational
schemas and common ORM models.

Implementation work:

- Generalize the SQL AST and persisted `ForeignKeyDefinition` from one column to
  ordered child and referenced column lists.
- Parse equivalent column-level and table-level forms, including optional names:

  ```sql
  customer_id INTEGER REFERENCES customers(id)

  CONSTRAINT fk_line_order
    FOREIGN KEY (order_id, line_no)
    REFERENCES order_lines(order_id, line_no)
  ```

- Require equal non-zero column counts and compatible types/collations. Bind to
  stable column and key ids rather than names alone.
- Require the referenced tuple to match an enforced primary or unique candidate
  key. Do not accept a merely indexed, non-unique tuple.
- Implement logical composite primary/unique constraints and composite unique
  indexes as prerequisites where the engine does not already enforce them.
- Create or reuse a supporting child index with columns in constraint order.
  Track ownership so dropping a foreign key removes only an automatically owned
  index that has no other dependency.
- Implement SQL `MATCH SIMPLE` null behavior first: if any child component is
  null, that row does not require a matching parent. Document `MATCH FULL`,
  `MATCH PARTIAL`, and deferred constraints separately until supported.
- Extend actions to explicit `ON DELETE` and `ON UPDATE` definitions. Target:
  `NO ACTION`, `RESTRICT`, `CASCADE`, `SET NULL`, and, after defaults are stable,
  `SET DEFAULT`.
- Treat `NO ACTION` as immediate behavior until deferred constraints exist, but
  keep its metadata value distinct from `RESTRICT`.
- Validate action prerequisites at DDL time. `SET NULL` requires nullable child
  columns; `SET DEFAULT` requires defaults whose resulting tuple can be checked.
- Apply parent-key updates as one statement-level referential operation. A
  composite cascade must never expose a half-updated tuple.
- Detect cycles, self-reference, repeated rows, maximum cascade depth, and
  multiple cascade paths deterministically. Either complete the statement once
  per affected row or fail it atomically with a specific diagnostic.
- Coordinate referential actions with triggers and checks through one mutation
  graph. Define observable ordering and prohibit recursion from bypassing limits.
- Extend referential locking/conflict rules to composite tuples and every action.
  Test concurrent child insert versus parent delete/update, competing cascades,
  parent-key uniqueness changes, retries, and rollback so no committed schedule
  can leave an orphan.
- Update dependency tracking for table/column/key/index rename and drop.
- Preserve old single-column metadata and behavior through serializer and client
  upgrades.

Acceptance criteria:

- Column-level and table-level single-column constraints enforce identically.
- Composite child tuples reference composite primary and alternate keys in the
  declared order and follow documented null semantics.
- Every advertised delete and update action is atomic for single, composite,
  self-referencing, chained, cyclic, and trigger-involved cases.
- Invalid parent updates remain blocked when no update action permits them.
- Rename, drop, reopen, backup/restore, schema scripting, and migration round
  trips retain key order, names, actions, and support-index ownership.
- Table, column, key, and constraint ids remain unchanged across reopen, rename,
  archive round trips, and direct/REST/gRPC schema transport.
- Metadata exposes one constraint row plus ordered child/parent mapping rows; it
  does not flatten composite columns into an ambiguous comma-separated string.
- Unsupported match modes and deferrability clauses fail explicitly and remain
  distinct matrix rows.

### 4. Complete The Practical `ALTER TABLE` Surface

Goal: support the schema evolution operations needed by applications and the
first-party EF Core provider without corrupting dependencies or losing data.

Direct metadata operations to target:

- `ADD CONSTRAINT` for check, unique, primary-key, and foreign-key definitions
  when existing data validates.
- `DROP CONSTRAINT` for every named constraint kind with dependency checks.
- `ALTER COLUMN ... SET DEFAULT` and `DROP DEFAULT`.
- `ALTER COLUMN ... SET NOT NULL` and `DROP NOT NULL`.
- Rename constraint and rename index. `RenameIndex` is already a first-party EF
  migration operation that must stop failing as the practical migration surface
  expands.

Rewrite-backed operations to target:

- Change a column type with an explicit safe-conversion policy and an optional
  CSharpDB-supported conversion expression when implicit conversion is unsafe.
- Change collation while rebuilding every affected index, key, check binding,
  and foreign-key compatibility relationship.
- Add a non-nullable/defaulted column to a populated table.
- Drop or reorder physical columns while preserving stable logical ids where
  feasible.
- Change primary/unique key shape, including composite keys.
- Apply constraint changes whose validation or storage layout requires a full
  scan.

Reusable table-rewrite design:

- Build one `TableRewritePlan` that records source schema, target schema,
  conversion expressions, validation steps, dependency reconstruction, and
  commit metadata.
- Route the current drop-column implementation through this mechanism, or harden
  it to the same atomicity and recovery contract, before using it as the basis
  for any additional rewrite-backed operation.
- Acquire an explicit write/exclusive schema lock for the first implementation.
  Online DDL is not required for this milestone.
- Create shadow storage, stream and convert rows with bounded memory, validate
  the target schema, rebuild indexes and constraints, preserve the identity
  high-water mark, and refresh/invalidate statistics.
- Atomically swap the table root and catalog state through the WAL. On failure,
  retain the old root; on recovery, deterministically finish or remove orphaned
  shadow state.
- Preserve triggers, views, diagrams, validation rules, inbound/outbound foreign
  keys, index ownership, Admin metadata, and schema-script signatures.
- Preflight all dependencies before copying rows. Fail early when a view,
  trigger, constraint, or provider operation cannot be rewritten safely.
- Expose progress and cancellation only after cancellation can leave a clean,
  recoverable state.
- Invalidate parsed statement, plan, catalog, prepared-command, and Admin schema
  caches at the atomic schema-version change.

Acceptance criteria:

- A failed conversion, check, uniqueness test, foreign-key validation, cancel,
  or simulated crash leaves the original table readable and unchanged.
- A successful rewrite preserves all rows, identity continuity, indexes,
  triggers, views, constraints, schema metadata, and inbound references.
- Reopen and WAL recovery tests cover failures before copy, during copy, during
  index rebuild, before swap, and immediately after swap.
- EF-generated up/down migrations for every newly supported operation execute
  against empty and populated databases.
- Unsupported alterations identify the exact unsupported change and never
  generate incomplete SQL.

### 5. `UNION ALL` And Compound-Query Semantics

Goal: add duplicate-preserving compound queries and make the complete compound
surface predictable for hand-written and generated SQL.

Implementation work:

- Add an explicit set quantifier to the AST. `UNION` continues to mean
  `UNION DISTINCT`; `UNION ALL` selects bag semantics.
- Parse mixed `UNION`, `UNION ALL`, `INTERSECT`, and `EXCEPT` chains with tested
  precedence, parentheses, CTE, subquery, view, `ORDER BY`, `LIMIT`, and `OFFSET`
  behavior.
- Bind every branch to the same column count and a documented common type and
  collation. Result column names come from the first branch.
- Preserve duplicates and SQL null values exactly for `UNION ALL`; do not route
  it through the existing distinct hash set.
- Add an append/concatenate operator that can stream the left input followed by
  the right input when there is no global operation requiring materialization.
- Allow an outer limit to stop branch consumption early when semantics permit.
  A global order, distinct operation, or later set operation may still require
  materialization or sorting.
- Support row and batch execution consistently and report branch/append nodes in
  physical plans and actual metrics.
- Add overflow, cancellation, disposal, and memory-bound tests for large and
  deeply nested compounds.

Acceptance criteria:

- `UNION ALL` preserves duplicate rows, duplicate nulls, and first-branch column
  names. It promises no output order without a final `ORDER BY`, even if a
  streaming implementation currently visits the left input first.
- Mixed set-operation precedence and parenthesized overrides match the published
  grammar.
- Type/collation conflicts fail during binding with both branch positions in the
  error.
- EF Core `Concat()` SQL snapshots execute with expected duplicate behavior;
  `Union()` retains distinct behavior.
- Streaming plans do not materialize both branches when no semantic requirement
  demands it.

### 6. Window Functions In Functional Tiers

Goal: support the window patterns most often generated by analytics queries and
ORM translations without promising the entire SQL window specification at once.

Tier 1 syntax and functions:

- Parse `OVER (...)`, `PARTITION BY`, window-local `ORDER BY`, and the first
  explicit `ROWS` frame forms.
- Implement `ROW_NUMBER`, `RANK`, and `DENSE_RANK`.
- Implement `COUNT`, `SUM`, `AVG`, `MIN`, and `MAX` as window aggregates.
- Define SQL-consistent default frames for windows with and without ordering,
  including peer handling. Do not approximate a peer-aware default with a
  different `ROWS` frame.
- Support common frames beginning with whole partition and unbounded-to-current
  row, then bounded preceding/following only after boundary tests are complete.

Tier 2 functions and frames:

- Add `LAG`, `LEAD`, `FIRST_VALUE`, and `LAST_VALUE`, including offset/default
  validation and null behavior.
- Expand bounded `ROWS BETWEEN` frames and combinations of current row,
  unbounded boundaries, and numeric offsets.
- Add named window clauses only if they materially reduce generated or
  hand-written SQL duplication.
- Keep `RANGE` offsets, `GROUPS`, frame exclusion, and vendor-specific null
  options as separate matrix rows until implemented exactly.

Execution work:

- Implement an explicit logical phase order: `FROM`/`WHERE`, grouping and
  `HAVING`, window evaluation, projection, `DISTINCT`, then final
  `ORDER BY`/`LIMIT`/`OFFSET`. Bind aliases and aggregate-plus-window references
  according to that order rather than treating windows as scalar calls.
- Restrict window calls to valid SELECT and ordering contexts. Reject them in
  `WHERE`, grouping keys, `HAVING`, defaults, checks, and other pre-window
  phases with a clear binding error.
- Add a window plan node/operator that groups compatible functions by partition,
  order, and frame so sorts and partition scans can be shared.
- Reuse an existing compatible input order when proven; otherwise make the sort
  explicit in diagnostics.
- Preserve SQL collation, null ordering, and peer semantics. When output
  determinism requires a tie-breaker, document that the caller must provide it
  rather than inventing a hidden user-visible order.
- Bound partition buffering and implement spill-to-disk before advertising
  large/unbounded partitions as production-safe.
- Keep row and batch output identical and account for sort, buffer, partition,
  frame, and spill work in plan metrics.

Acceptance criteria:

- Ranking results are correct for empty input, one row, ties, null sort keys,
  multiple partitions, descending order, and collations.
- Aggregate windows distinguish partition totals, running frames, peer groups,
  and ordinary grouped aggregates.
- Aggregate-plus-window queries, projected aliases, outer ordering, `DISTINCT`,
  limits, and invalid pre-window references follow the published logical phase.
- Multiple compatible window expressions share sort/partition work; incompatible
  definitions appear as distinct plan stages.
- Large-partition tests prove the memory bound and spill cleanup after success,
  cancellation, and failure.
- Each unsupported function, frame, or clause has its own matrix row and a
  precise diagnostic. Supported window rows remain supported even while the
  category contains other unsupported rows.

### 7. Function Catalog And Coverage

Goal: turn the existing useful but informal function set into a documented,
qualified contract and close gaps based on real generated SQL.

Implementation work:

- Inventory every built-in scalar and aggregate name and alias currently handled
  by `DbBuiltInScalarFunctions` and the aggregate binder.
- Introduce one function descriptor/registry containing canonical name, aliases,
  kind, argument counts, accepted types, return-type rule, null behavior,
  volatility/evaluation scope (`immutable`, `statement_stable`, or `volatile`),
  side-effect/trust classification, collation behavior, batch support, and
  default/check eligibility.
- Make parsing, binding, constant folding, scalar execution, batch execution,
  diagnostics, metadata, and documentation consume that shared descriptor where
  practical.
- Preserve existing aliases while documenting whether they follow SQL-standard,
  SQLite-like, Access-like, or CSharpDB-specific semantics.
- Prioritize missing functions from captured first-party EF Core and ADO.NET
  workloads. Rank gaps by query frequency and by whether they block a complete
  application pattern.
- Qualify string, comparison, numeric, date/time, conversion, conditional, and
  aggregate families before opening new families solely for breadth.
- Define invariant-culture conversion, overflow, divide-by-zero, invalid date,
  precision, rounding, current-time, time-zone, and null-propagation behavior.
- Constant-fold only immutable functions, evaluate statement-stable defaults
  once per statement, and exclude volatile or side-effecting callbacks from
  checks and schema expressions.
- Ensure optimized/batch paths either implement identical semantics or use a
  tested scalar fallback. A performance optimization must not create a second
  function dialect.
- Expose built-ins through a `sys.functions` catalog including signature and
  determinism fields; keep user-registered callbacks clearly distinguished.
- Generate the public function table from the registry or a checked manifest so
  aliases and arity cannot drift from code.

Acceptance criteria:

- Every callable built-in has a documented signature, return rule, null rule,
  and stable test id.
- Scalar, batch, constant-folded, prepared, direct, and remote evaluation agree
  for valid values, boundary values, nulls, and errors.
- Functions allowed in defaults and checks are selected from explicit registry
  metadata, not ad hoc name lists.
- Added functions correspond to a recorded compatibility workload or an explicit
  dialect decision.
- Unknown names and invalid arity/type combinations produce binding errors that
  name the function and expected signatures.

### 8. Generated SQL And ORM Compatibility

Goal: qualify concrete framework workloads and make the first-party EF Core
provider emit only SQL and schema operations CSharpDB supports.

Compatibility corpus:

- Capture `ToQueryString()` golden snapshots and executable fixtures for every
  supported EF Core query pattern. Snapshot success alone is insufficient; each
  emitted statement must execute and validate its result.
- Expand beyond CRUD to grouping/aggregates, correlated subqueries, null
  semantics, string/date/math translations, `Union`, `Concat`, pagination,
  projections, joins, optional relationships, owned entities, many-to-many join
  tables, composite keys, and filtered/top-N include patterns.
- Give EF `ExecuteUpdate` and `ExecuteDelete` independent corpus and matrix rows;
  they bypass the custom `SaveChanges` command batch and exercise different SQL
  generation and mutation paths.
- Add an ADO.NET/Dapper-style corpus covering named parameters, repeated
  parameters, scalar queries, multi-row reads/writes, transactions, reader
  schema, and common identifier quoting.
- Use SQLite differential runs only for cases whose semantics are intentionally
  aligned, and retain direct expected-result assertions for CSharpDB-specific
  behavior.
- Pin the first qualification baseline to EF Core 10.0.x and the repository's
  current 10.0.8 package. Require patch upgrades to replay the full corpus; add a
  new qualification row before claiming a later minor or major line.

Provider work:

- Add provider-specific translations only when the target CSharpDB function or
  SQL feature is fully implemented and registered.
- Support quoted and reserved identifiers through a single delimiter/escape
  contract rather than rejecting every identifier outside a simple alphanumeric
  pattern.
- Ensure Boolean predicates, null compensation, casts, aliases, parameter types,
  pagination, and correlated references match the engine binder.
- Map EF `Union()` and `Concat()` to the distinct and all variants respectively.
- Use windows for provider patterns such as top-N per group or filtered include
  only after the required frame/order semantics are qualified.
- Replace the zero-writable-column insert workaround with `DEFAULT VALUES` and
  generalize the current single-integer-identity result side channel into an
  additive structured result containing generated column ids/names, types, and
  values for each affected row. Return identity and default-generated values
  from the executing statement's transaction; do not issue an unsafe keyed
  readback. Treat user-facing SQL `RETURNING` as a separate matrix item.
- Lift model-validator rejections for defaults, checks, optional relationship
  actions, alternate/composite keys, composite indexes/FKs, and altered columns
  only after core engine, migration, persistence, and metadata tests pass.
- Expand migrations to generate the milestone-4 syntax and table rebuilds. Run
  real add/update/script/up/down flows against empty and populated databases.
- Keep idempotent migration-script generation explicitly unsupported in the
  first slice, with a matrix row and negative test, unless a correct history and
  locking design is promoted into this milestone.
- Make every unsupported model operation fail before SQL execution and include
  the property/constraint and unsupported behavior in the error.
- Defer reverse engineering (`dotnet ef dbcontext scaffold`) until metadata is
  complete enough to reproduce keys, checks, defaults, indexes, relationships,
  identity, types, and collations without loss.

Acceptance criteria:

- Golden generated SQL is reviewed, executes, and returns behaviorally correct
  results on direct CSharpDB.
- Replay applicable provider-emitted SQL through raw ADO.NET, HTTP, and gRPC
  clients to prove engine parity. This does not advertise remote EF connections;
  the first-party provider remains embedded-only unless a separate remote
  provider design is approved.
- Multi-row inserts can return multiple identity/default-generated values with
  stable row/column association and without an extra race-prone query.
- EF migration up/down and generated scripts preserve data and model shape for
  every advertised operation.
- No composite or unsupported constraint can be silently omitted from
  `CreateTableOperation` or another generated migration.
- The provider README and public matrix list qualified EF versions and query,
  mutation, migration, and scaffolding status independently.

### 9. Metadata Catalogs And Introspection

Goal: make schema state complete, normalized, and consistent enough for tools to
understand the capabilities implemented above.

SQL catalog work:

- Extend `sys.columns` with canonical default SQL and any stable generated-value
  classification introduced by this plan.
- Add normalized catalogs for constraints, check constraints, key constraints,
  key columns, foreign-key columns, and functions.
- Keep `sys.foreign_keys` as one row per constraint and add ordered mapping rows
  in `sys.foreign_key_columns`. Preserve old single-column columns or aliases for
  compatibility, but do not encode composite tuples as comma-separated text.
- Include constraint names/ids, table ids, column ordinals, referenced key,
  match mode, delete rule, update rule, enforcement state, and support-index
  ownership where applicable.
- Preserve and extend the existing ordered-per-index-column `sys.indexes`
  contract for composite indexes; add a new catalog only if it supplies a
  distinct, documented shape.
- Define compatibility behavior for the existing scalar
  `sys.tables.primary_key_column` when a logical primary key is composite. Keep
  it for single-column callers and return a documented null/sentinel for
  composite keys while normalized key catalogs carry the complete shape.
- Consider a targeted `information_schema` compatibility layer for `tables`,
  `columns`, `table_constraints`, `key_column_usage`, and
  `referential_constraints`. Do not claim a complete `information_schema` until
  every required view and semantic is present.
- Make catalog reads observe transaction and schema-version rules consistently;
  invalidate catalog caches after committed DDL.
- Define one user/system/internal/temporary object classification and apply it
  consistently to SQL catalogs, direct schema APIs, ADO.NET metadata, Admin,
  and remote clients. Include EF history/lock tables explicitly; do not rely on
  a leading underscore as an undocumented visibility rule.

ADO.NET and client work:

- Expand `GetSchema()` collections with `DataSourceInformation`, `DataTypes`,
  `Restrictions`, `ReservedWords`, normalized index columns, constraints,
  checks, keys, triggers, and functions where the ADO.NET contract is clear.
- Populate `COLUMN_DEFAULT`, key ordinals, `UPDATE_RULE`, `DELETE_RULE`, match
  mode, identity, precision/scale/size, and restriction columns accurately.
- Add binder/planner-produced result-column metadata containing output alias,
  logical type/nullability, expression/read-only state, base table/column
  lineage, and key/unique/identity flags. Persisted `ColumnDefinition` alone
  cannot describe joins, aliases, expressions, or compound results.
- Expand `DbDataReader.GetSchemaTable()` with base catalog/table/column
  provenance, key/unique/identity/read-only flags, size, precision, scale, and
  nullability. Implement `IDbColumnSchemaGenerator` so `GetColumnSchema()`
  exposes the same contract.
- Carry structured result-column metadata through `QueryResult` and every
  transport. Remove remote type inference from returned row values so empty and
  all-null result sets retain their declared types.
- Propagate additive schema fields through direct client models, REST DTOs,
  protobuf, transport mappers, `RemoteDatabaseSession`, DevOps schema scripts,
  schema comparison, archives, Admin designers, and MCP descriptions.
- Preserve forward compatibility for older clients: additive fields must have
  safe defaults, and version negotiation must prevent a client from rewriting a
  schema shape it cannot represent.

Acceptance criteria:

- The same committed schema produces semantically identical catalog results
  through direct SQL, ADO.NET, HTTP, and gRPC.
- Composite keys, indexes, and foreign keys retain ordered rows and stable
  constraint identity after reopen and rename.
- A schema script generated from catalogs recreates defaults, checks, keys,
  foreign-key actions, indexes, and collations without silent loss.
- Metadata restrictions and result columns conform to documented ADO.NET shapes
  and refresh after DDL without reconnecting where supported.
- Direct and remote readers return identical declared types and lineage for
  empty, all-null, aliased, joined, expression, aggregate, and compound results.
- Object visibility is identical across catalogs and providers for user,
  system, internal, temporary, and EF-owned objects.
- Admin and DevOps tools either round-trip a new feature or explicitly show it
  read-only; they never save an older partial representation over it.

### 10. Physical `EXPLAIN`, Query Diagnostics, And `EXPLAIN ANALYZE`

Goal: show the plan CSharpDB actually selects and explain how estimated behavior
compares with real execution.

Physical plan foundation:

- Preserve `EXPLAIN ESTIMATE FOR` syntax and its exact legacy contract: column
  names, order, types, nullability, decision tokens, statistics-state tokens,
  and non-execution behavior. Keep it an estimate-input compatibility view even
  if shared plan data improves its internals; introduce physical-plan output
  under the new `EXPLAIN` form rather than silently redefining the legacy form.
- Add a sentinel user-function proof that fails if estimate-only explain invokes
  the target expression. Parser coverage alone does not prove non-execution.
- Introduce a shared physical-plan descriptor with stable node identity that is
  created by the same decisions that select scans, indexes, joins, aggregates,
  compounds, windows, CTE handling, sorts, limits, and fast paths.
- Require every construction path—external scans, adaptive plans, materialized
  paths, row/batch operators, synchronous lookup, and scalar fast paths—to emit
  a descriptor. A path without a descriptor cannot be marked supported for
  physical explain.
- Eliminate or reconcile the separate estimate-only AST walker so diagnostics
  cannot claim an operator or access path different from the one execution uses.
- Include logical operation, physical operator, parent/child relationship,
  target object/index, predicates, projections, estimated rows/cost, statistics
  source/state, ordering, memory/spill expectation, and plan-cache information.
- Offer a stable tabular contract. A structured JSON representation may be
  added for Admin and tooling, but it must be versioned and derived from the
  same plan tree.
- Add a standard-friendly `EXPLAIN` form while retaining the current estimator
  alias. Freeze the exact grammar and DML rejection behavior in parser tests.

Actual execution instrumentation:

- Add optional per-node counters around both `IOperator` and `IBatchOperator`.
  Instrument synchronous lookup/scalar and materialized fast paths as well, or
  route them through an equivalent measured plan node.
- Freeze the first profile row schema and metric semantics before coding.
  Distinguish rows consumed from children from rows emitted to the parent;
  define `loops` as operator-open count; report unavailable metrics as null;
  report an opened empty node as zero rows; and identify never-opened nodes
  separately.
- Capture rows consumed/emitted, loops, batches, inclusive and self/exclusive
  elapsed time, first-row time, page reads, bytes where measurable, peak memory,
  spill bytes/runs, planned operator, executed operator, and adaptive decisions.
- Add per-node `status` and `stop_reason` values such as `exhausted`,
  `parent_stop`, `not_opened`, `cancelled`, and `failed`. A successful `LIMIT`
  is statement-complete even when its child correctly stops with `parent_stop`.
- Compute an estimated-to-actual row ratio without dividing by zero and surface
  large cardinality errors in diagnostics.
- Keep instrumentation disabled by default and benchmark its disabled and
  enabled overhead.
- Reset counters on every prepared/cached execution and keep instrumentation
  wrappers out of later normal executions. Preserve stable node ids across an
  adaptive switch while exposing planned and executed operators separately.
- Ensure cancellation, exceptions, disposal, and partial operator opens finalize
  counters predictably and mark genuinely incomplete execution.

`EXPLAIN ANALYZE` behavior:

- Start with read-only `SELECT`, `WITH`, and compound queries. Reject DDL and DML
  explicitly in the first release rather than executing side effects
  unexpectedly.
- Execute the target exactly once and drain it to completion so actual row
  counts are meaningful. Return plan/metric rows, not the target query rows.
- Make the side-effect contract prominent: the target query executes once, and
  registered functions and external reads follow ordinary SELECT invocation
  semantics during that execution. This is profiling, not simulation; callers
  must not use it on a supposedly read-only query whose callbacks have undesired
  side effects.
- Guarantee that the root actual row count equals the row count of ordinary
  execution for the same snapshot and parameters.
- Put redaction in the shared diagnostic serializer and apply it to legacy
  estimates, physical plans, and analyzed plans. Redact literals, parameter
  values, paths, and external connection details by default in shareable,
  remote, and Admin output while preserving enough shape to troubleshoot. Any
  trusted-local detailed mode must be explicit and unavailable remotely.
- Add direct, ADO.NET, HTTP, and gRPC contract tests. The explain result is a
  regular SQL rowset and should not require a transport-specific endpoint.
- For the first result contract, propagate cancellation/execution failure as an
  error and retain incomplete metrics only in internal diagnostics/logging.
  Returning both an error and a partial profile requires a separately versioned
  result envelope and is not part of the regular-rowset slice.
- Cover prepared-parameter rebinding, classification, script splitting, ADO.NET
  prepared commands, REST/gRPC mapping, and Admin statement builders. For
  sharded use, support route-bound explain on one selected shard and reject
  fan-out profiling until a distributed profile contract exists.
- Update Admin with distinct `Estimate`, `Physical Plan`, `Profile`, and
  `Refresh Stats` actions. Update public and embedded help so the current
  statistics-refresh action is never confused with `EXPLAIN ANALYZE`.

Acceptance criteria:

- Representative scan, index seek, join, aggregate, set, CTE, window, sort,
  limit, and fast-path queries report the operator path that actually ran.
- Existing `EXPLAIN ESTIMATE FOR` tests and non-execution guarantee remain valid.
- `EXPLAIN ANALYZE` actual rows and loops are correct for empty inputs, filters,
  joins, aggregates, windows, limits, compounds, and repeated operator opens.
- Adaptive/cached executions retain stable planned node identity, show the
  executed choice, reset counters, and do not leak profiling into later queries.
- Cancellation and failure propagate the SQL error, retain an explicitly
  incomplete profile only for internal diagnostics, and release all resources
  and spill files.
- Metric collection has measured, documented overhead and does not affect query
  results or transaction semantics.
- Public SQL docs and the compatibility matrix distinguish estimate-only,
  physical-plan, and analyzed-execution capabilities.

## Compatibility And Rollout

- Land new schema fields additively and keep old readers/writers in the test
  matrix. A client that cannot represent a new constraint must not be allowed to
  overwrite it through schema-edit operations.
- Continue accepting current SQL spellings and the existing
  `EXPLAIN ESTIMATE FOR` contract.
- Make new behavior opt-in through syntax; avoid compatibility switches that
  cause the same statement to mean different things in two databases.
- If a file-format feature requires a minimum reader version, record it in
  database metadata and return a precise upgrade error instead of misreading it.
- Publish matrix changes with a `since` version and preserve prior released
  manifests or release tags so users can compare upgrade targets.
- Treat provider qualification separately from engine support. A SQL feature can
  be supported for hand-written SQL while an EF translation remains partial.
- Roll out high-risk features in narrow tiers: immediate foreign keys before
  deferrable behavior, `ROWS` windows before advanced frames, read-only
  `EXPLAIN ANALYZE` before DML profiling, and offline transactional DDL before
  online DDL.
- Keep unsupported cases as explicit negative tests through rollout so a future
  parser expansion cannot accidentally turn a clear rejection into partial,
  unsafe execution.

## Out Of Scope For The First Compatibility Release

- Blanket ANSI/ISO SQL conformance or compatibility with an arbitrary vendor
  dialect.
- Automatic rewriting of SQL Server, PostgreSQL, MySQL, Oracle, or SQLite SQL.
- Deferred foreign-key/check enforcement and full `MATCH FULL`/`MATCH PARTIAL`
  semantics unless promoted into a later explicit milestone.
- Online, non-blocking, or distributed schema changes.
- Every SQL window function, `RANGE` offset, `GROUPS`, and frame-exclusion form.
- Computed/generated columns, sequences, schemas, SQL procedure DDL/catalog
  integration, or a full `information_schema`, unless separately prioritized by
  the workload corpus. Existing client-managed procedure features remain
  separate and are not removed by this exclusion.
- Idempotent EF migration scripts until a history/locking design is promoted
  from its explicit unsupported matrix row.
- `EXPLAIN ANALYZE` for statements that mutate data or schema.
- Deterministic wall-clock timings in explain output; row/loop correctness is a
  contract, exact elapsed time is diagnostic.
- General EF Core reverse engineering before metadata can round-trip the model.
- Claiming support for every ORM merely because a few generated statements
  parse.

## Assumptions

- Embedded/direct use remains a first-class deployment mode, and daemon clients
  execute the same core parser, planner, catalog, and engine semantics.
- The existing WAL and transaction layer can be extended to coordinate an
  atomic table-root/catalog swap; if not, that storage prerequisite becomes a
  blocking design milestone for rewrite-backed `ALTER TABLE` work.
- Logical composite candidate keys can be enforced with existing or generalized
  unique indexes without requiring the storage engine's physical primary key to
  become composite immediately.
- The first-party EF Core provider is the primary generated-SQL compatibility
  target. ADO.NET/Dapper-style SQL is the secondary workload corpus.
- Existing function names and aliases are compatibility surface even when a
  canonical registry later prefers a different spelling.
- The static `www` site can accept a deterministic generated HTML artifact and
  machine-readable JSON checked into the repository.
- Public compatibility evidence can use stable test ids without exposing
  internal test implementation details.
