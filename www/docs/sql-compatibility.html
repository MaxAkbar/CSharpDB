EF migrationpartial

Limitations

  • Only INTEGER-to-REAL and REAL-to-INTEGER are accepted. INTEGER values must remain within the exactly representable inclusive range -2^53 through 2^53; REAL values must be finite, integral, and within the signed 64-bit INTEGER range. An incompatible stored value leaves the original root and schema unchanged. The altered column cannot participate in a primary/unique key, ordinary index, or incoming/outgoing foreign key. The conservative preflight also rejects any view that references the table, any trigger owned by the table, a cross-table trigger that references the column, and an applicable validation rule. Affected-index rebuild and dependency rewriting are not implemented. TEXT/BLOB conversions remain unsupported. A persisted default must validate for the target type, so DROP DEFAULT and a later compatible SET DEFAULT may be required. The generated multi-command default/type/nullability sequence must execute in one migration transaction so a conversion failure restores every facet.

Intentional deviations

  • Successful conversion streams rows through a transactional shadow root, preserves each physical row ID and indexes on other columns, revalidates CHECK constraints, swaps the catalog root atomically, and persists across reopen. EF Core infers the old store type independently of the target model column and orders DROP DEFAULT, TYPE, SET DEFAULT, and nullability changes when those facets change together.

Limitations

  • SET COLLATION and DROP COLLATION apply only to TEXT columns. Ready ordinary and unique SQL indexes that inherit the column collation are rebuilt; indexes with an explicit collation and indexes on other columns retain their roots. Primary/unique key constraints, incoming/outgoing foreign keys, full-text or collection dependencies, and non-ready indexes remain unsupported.
