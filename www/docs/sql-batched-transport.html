SQL Batched Row Transport Design (Paused)

Status At A Glance

Status (March 2026): Paused. This document should be read as paused design context plus retained-in-repo groundwork, not as a shipped feature and not as an active implementation task.

What is already done and retained:

  • RowBatch and IBatchOperator exist in src/CSharpDB.Execution/IOperator.cs
  • BatchToRowOperatorAdapter exists in src/CSharpDB.Execution/BatchTransport.cs
  • QueryResult can hold batch-native roots directly and preserve partial consumption correctly
  • RowSelection, IFilterProjectionBatchPlan, BatchPlanCompiler, and SpecializedFilterProjectionBatchPlan exist as internal scaffolding
  • Several operators can already produce or consume batches internally

What is not done:

  • The SQL executor is not broadly planner-driven around true end-to-end batch transport
  • The executor still fundamentally centers on row-by-row IOperator transport
  • The generic expression evaluator contract has not been replaced
  • There is no stable broad production rollout

Summary

CSharpDB's executor still fundamentally transports one logical row at a time through IOperator. That is true even after the recent compact scan, compact payload, aggregate, lookup, and join optimizations. Some operators now batch internal work, but the public executor contract is still:

  • MoveNextAsync()
  • one DbValue[] Current
  • optional row-buffer reuse via ReusesCurrentRowBuffer

Why This Exists

Recent work proved three things:

  • compact table-scan and compact payload projection paths benefit from local batching on expression-heavy work
  • generic projection and filter/projection operators can also benefit from batched expression shaping
  • the current gains are bounded because every operator boundary still collapses back to single-row transport

Problem Statement

The current IOperator shape creates several costs:

  • every operator boundary requires row-by-row control flow
  • expressions are still invoked one row at a time even when upstream work is scan-oriented
  • filters and projections cannot share a stable batch buffer across multiple operators
  • joins and sorts still frequently re-materialize rows into per-row arrays
  • any future vectorized work has to be implemented as operator-local special cases

Goals

  • introduce a true internal batch transport between operators
  • reduce per-row control-flow overhead in scan-heavy execution
  • reduce DbValue[] churn across operator boundaries
  • make expression, filter, and projection execution batch-aware
  • give joins, aggregates, and sorts a common batch-shaped input model
  • preserve existing SQL semantics, ordering behavior, and null semantics

Non-Goals

  • changing the on-disk row format
  • replacing DbValue
  • adding SIMD-specific logic in the first phase
  • forcing every operator to become batch-native immediately
  • breaking the public query surface

Current Status

The first retained benchmark signal for the specialized plan is positive on the internal operator path:

  • projection-only batch plan: about 378.8 us down to 345.5 us over 16,384 rows
  • filter + projection batch plan: about 221.4 us down to 208.8 us over 16,384 rows

That is enough to justify keeping the specialized-plan direction, but not enough yet to route production queries through it broadly.

Pause Decision

As of March 22, 2026, this workstream is paused. The retained groundwork is useful and should stay. The small and medium follow-on experiments have been exhausted. Repeated production rollouts did not produce stable enough wins.

The project should not come back here for:

  • more adapter experiments
  • more alternate delegate shapes
  • more narrow planner hooks over the current evaluator model
  • more compact-operator-specific special cases

Revisit Gate

Only reopen this work when all of the following are true:

  1. there is explicit willingness to do a deeper SQL executor rewrite rather than another bounded experiment
  2. the rewrite target is a new evaluator contract, not another wrapper over Func<DbValue[], DbValue>
  3. benchmark success criteria are agreed up front for scan-heavy production shapes
  4. the rollout starts behind a narrow gate and is measured against the current stable executor baseline

Restart Direction If Reopened

1. Add a batch-native internal contract

Add a new internal transport alongside IOperator, rather than replacing it immediately. Recommended shape: IBatchOperator, RowBatch, BatchExecutionResult.

2. Use a flat row-major batch first

Recommended first representation:

  • one flat DbValue[] storage buffer sized capacity * columnCount
  • Count, ColumnCount
  • optional active row mask or row count after filtering

3. Make ownership explicit

Recommended rule: batches are reusable by default. Consumers that retain rows beyond the next MoveNextBatchAsync() must copy.

Retained Types And Compatibility Seams

RowBatch

Already present: DbValue[] Values, int Count, int Capacity, int ColumnCount. Supports GetRowSpan, CopyRowTo, reset/reuse.

IBatchOperator

Already present: OutputSchema, ReusesCurrentBatch, CurrentBatch, OpenAsync, MoveNextBatchAsync.

Adapters

BatchToRowOperatorAdapter exists. Broad RowToBatchAdapter expansion was tried and backed out because it erased gains on join-expression and scan-projection workloads.

Tried And Backed Out

The following approaches were implemented, benchmarked, and reverted because results were not good enough:

  1. Broad RowToBatch adapter expansion — scan-projection results mixed to worse, join-expression rows regressed
  2. Batch-aware expression evaluator seam — join-expression benchmarks regressed
  3. Parallel compiled batch-expression trees — join expression projection rows regressed enough to erase scan wins
  4. Additional operator-local transport experiments — no stable broad win
  5. ReadOnlySpan<DbValue> compiled delegate evaluators — scan and join regressions
  6. Narrow bytecode evaluator + selection-vector filtering — scan projection regressed clearly
  7. Planner-routed delegate-backed batch plan — join regressions outweighed scan improvements
  8. Planner-routed specialized batch plan — broad production routing not good enough
  9. Narrow specialized-plan rollout in compact operators — still not compelling enough
  10. Direct row-specialized compact fast-path rollout — expression path still slower on important scan-heavy shapes

Memory Model

The first implementation should prefer reusable fixed-capacity batches, predictable buffer lifetimes, and low-copy flow between producer and consumer.

Batch size should be configurable internally, with a conservative first default (e.g., 64 rows for projection-heavy operators, possibly 128 or 256 for simpler scan/filter paths).

Benchmark Gate If Reopened

Coverage should include:

  • full scan SELECT *
  • filtered scan SELECT *
  • filtered scan with expression projection
  • indexed range scan with expression projection
  • join with expression projection
  • grouped aggregate over scan-heavy input

Success criteria: lower allocation, fewer operator-local buffers, lower mean latency on 10K and 100K scan-heavy cases.

Risks

  • batch adapters may erase early gains if they are inserted too often
  • row-major batches may still leave some SIMD/vector opportunities unrealized
  • sort/join operators may need their own retained-batch copy strategy
  • debugging correctness regressions gets harder when a bug affects multiple rows per call

These are acceptable if the rollout stays phased.
