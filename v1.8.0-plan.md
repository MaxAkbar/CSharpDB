# CSharpDB v1.8.0 Release Plan

## Release Theme

`v1.8.0` should turn the current transport model into a real networked story while cleaning up package boundaries that are still carrying transition-era naming and compatibility layers.

Primary outcomes for this release:

- Make `Grpc` an actual supported transport in `CSharpDB.Client`
- Update the service-daemon documentation to present gRPC as a real supported path, not only future intent
- Formalize the package move from `CSharpDB.Core` to `CSharpDB.Primitives`
- Reduce `CSharpDB.Service` to a clearly deprecated compatibility package instead of a parallel long-term API
- Add repeatable benchmarks so transport and daemon performance can be measured and compared
- Stage `CSharpDB.Core` and `CSharpDB.Service` for removal in `v2.0.0`

## Goals

### 1. Ship real gRPC support in `CSharpDB.Client`

Today the client contract exposes `Grpc` in `CSharpDbTransport`, but the implementation still fails with a not-implemented path. `v1.8.0` should close that gap and make gRPC a first-class, documented transport.

Success criteria:

- `CSharpDB.Client` can connect to a daemon over gRPC and execute supported operations end to end
- gRPC configuration is discoverable through client options and daemon settings
- integration tests prove parity between direct execution and gRPC for the supported surface
- docs and samples stop describing gRPC as placeholder-only

### 2. Align daemon docs with the real transport story

The daemon roadmap already mentions gRPC in diagrams and architecture, but the implementation plan and client sections still lean heavily toward HTTP-only examples. `v1.8.0` should make the daemon docs consistent with the actual client/server transport matrix.

Success criteria:

- `docs/service-daemon/README.md` explicitly documents gRPC as a supported daemon transport for this release
- examples show `CSharpDB.Client` connecting over gRPC
- phase ordering, configuration, and deliverables are updated so the document matches the release plan

### 3. Finalize the package move from `CSharpDB.Core` to `CSharpDB.Primitives`

The repo already contains a compatibility story, but the release should make `CSharpDB.Primitives` the canonical package identity everywhere user-facing.

Success criteria:

- docs, package references, badges, and examples prefer `CSharpDB.Primitives`
- `CSharpDB.Core` is treated as a compatibility package, not the primary package name
- migration guidance is included in release notes and upgrade docs

### 4. Resolve the role of `CSharpDB.Service`

With `CSharpDB.Client` as the authoritative API and the daemon becoming the network host story, `CSharpDB.Service` should stop looking like an equal long-term surface area.

Success criteria:

- `CSharpDB.Service` is explicitly positioned as a deprecated compatibility facade
- no new database features are added to `CSharpDB.Service` unless required for compatibility forwarding
- docs steer new consumers to `CSharpDB.Client` or the daemon instead of `CSharpDB.Service`
- `v1.8.0` documentation and warnings make the `v2.0.0` removal timeline explicit

### 5. Add benchmarks for transport and daemon performance

`v1.8.0` should not rely on anecdotal performance claims. The release needs a benchmark story that lets the repo compare direct execution against daemon-backed access, especially gRPC.

Success criteria:

- repeatable benchmarks exist for representative read and write workloads
- results can compare `Direct`, `Http`, and `Grpc` where implemented
- benchmark runs distinguish warm and cold scenarios where that matters
- release notes can reference measured throughput and latency instead of assumptions

## Recommended Release Positioning

### Authoritative surfaces after `v1.8.0`

- `CSharpDB.Client`: authoritative application-facing client API
- `CSharpDB.Daemon`: long-running service host for remote and multi-client access
- `CSharpDB.Primitives`: canonical low-level package for shared types

### Compatibility-only surfaces after `v1.8.0`

- `CSharpDB.Core`: compatibility package forwarding users to `CSharpDB.Primitives`
- `CSharpDB.Service`: compatibility facade for older in-process hosts that still inject `CSharpDbService`

### Removal target after `v1.8.0`

- `v2.0.0`: remove `CSharpDB.Core` compatibility packaging once `CSharpDB.Primitives` migration has been in place for one release cycle
- `v2.0.0`: remove `CSharpDB.Service` once `CSharpDB.Client` and daemon-backed integration paths cover the supported migration scenarios

## Scope

### In Scope

- End-to-end gRPC transport support for `CSharpDB.Client`
- Daemon-side gRPC endpoint hosting and configuration
- Release documentation updates for gRPC usage
- Canonical package naming update from `CSharpDB.Core` to `CSharpDB.Primitives`
- `CSharpDB.Service` deprecation and migration messaging
- Benchmark coverage for direct and daemon-backed transport performance

### Out of Scope

- Full transport completion for `Tcp` and `NamedPipes`
- Removing `CSharpDB.Service` entirely in this release
- Large public namespace renames if package identity changes are enough for `v1.8.0`
- Advanced daemon features such as subscriptions, replication, or auth redesign unless required by gRPC delivery

## Workstreams

### Workstream 1: gRPC in `CSharpDB.Client`

Deliverables:

- Implement a real `Grpc` transport path in `CSharpDB.Client`
- Add transport configuration for endpoint, TLS/insecure local development, and channel setup
- Define the supported RPC contract for the client operations included in `v1.8.0`
- Add integration coverage for CRUD, SQL execution, schema inspection, and error handling over gRPC
- Update `src/CSharpDB.Client/README.md` and top-level docs to remove not-implemented messaging for gRPC where applicable

Notes:

- Prefer a clearly defined supported subset over claiming the full interface if some areas are not daemon-safe yet
- Unary request/response RPCs are enough for `v1.8.0`; streaming can remain future work unless it falls out naturally
- Error mapping must preserve the current client contract as closely as possible

### Workstream 2: Daemon gRPC Host

Deliverables:

- Add a gRPC server surface to the daemon
- Define how gRPC and HTTP coexist in daemon hosting and configuration
- Document port binding, endpoint selection, and local development defaults
- Add daemon integration tests that validate `CSharpDB.Client` against the hosted gRPC endpoint

Notes:

- The daemon README already advertises gRPC conceptually; this release must make the implementation plan match that claim
- If both HTTP and gRPC are supported simultaneously, configuration examples should show both clearly

### Workstream 3: `docs/service-daemon/README.md` Rewrite Pass

Required updates:

- Move gRPC from "architectural intent" to "release-supported transport"
- Update implementation phases so client SDK work explicitly includes gRPC, not only HTTP
- Add a `.NET` example using `CSharpDB.Client` with `CSharpDbTransport.Grpc`
- Review diagrams, configuration tables, deliverables, and priority labels for consistency
- Add migration guidance for users choosing between direct client usage, HTTP daemon access, and gRPC daemon access

### Workstream 4: `CSharpDB.Core` -> `CSharpDB.Primitives`

Deliverables:

- Audit docs and package guidance so `CSharpDB.Primitives` is always the primary recommendation
- Keep `CSharpDB.Core` only as a compatibility package during the transition window
- Publish upgrade guidance showing the package-reference change and any namespace expectations
- Update release notes and README references that still imply `CSharpDB.Core` is the primary package
- Add explicit `v2.0.0` removal messaging to compatibility docs and package metadata

Breaking-change guidance:

- Treat the package identity move as a documented breaking change for consumers that still reference `CSharpDB.Core` directly
- If namespaces remain `CSharpDB.Core` for compatibility, state that clearly so the change is understood as package-level first, namespace-level later if ever needed

### Workstream 5: `CSharpDB.Service` Deprecation Strategy

Recommendation for `v1.8.0`:

- Keep `CSharpDB.Service`, but freeze it as a compatibility facade
- Mark the package and key entry points as deprecated/obsolete where practical
- Do not add new first-class features directly to `CSharpDB.Service`
- Route any required behavior through `CSharpDB.Client`
- Remove `CSharpDB.Service` from primary getting-started guidance and package recommendations
- Mark `v2.0.0` as the intended removal release in docs and migration guidance

Why this approach:

- It reduces user confusion without forcing an immediate hard removal
- It matches the existing repo direction where `CSharpDB.Client` is already authoritative
- It leaves room to fully remove the compatibility layer in a later major release after one clear deprecation cycle

### Workstream 6: Benchmarks and Performance Evaluation

Deliverables:

- Add a benchmark project or benchmark suite for representative database workloads
- Measure direct client performance against daemon-backed HTTP and gRPC execution
- Capture both latency and throughput for core operations
- Include cold-start and warm-path scenarios for daemon access
- Document how to run benchmarks and where to publish baseline results for `v1.8.0`

Recommended benchmark scenarios:

- point lookup by primary key
- paged table browse
- simple filtered `SELECT`
- bulk insert/write transaction
- mixed read/write workload under concurrent clients
- daemon startup plus first-query latency

Notes:

- BenchmarkDotNet is the simplest fit for repeatable .NET-side measurement if no existing benchmark harness already exists
- Results should separate transport overhead from engine execution cost as much as possible
- Keep hardware and dataset assumptions documented so numbers are comparable over time

## Breaking Changes

Expected breaking changes or migration-impacting changes for `v1.8.0`:

1. `CSharpDB.Primitives` becomes the canonical package replacing `CSharpDB.Core` in user-facing guidance
2. `CSharpDB.Core` and `CSharpDB.Service` become explicitly deprecated with planned removal in `v2.0.0`
3. documentation and examples will stop treating `CSharpDB.Service` as a preferred dependency for new work
4. gRPC becomes an actual supported network transport, which may tighten endpoint/configuration validation for `Grpc` options that currently only pass shape checks

## Breaking Change Release Strategy

Recommended release approach: ship `v1.8.0` as a compatibility-backed transition release, not as a hard-removal release.

That means the repo should release the new direction in `v1.8.0`, but keep compatibility packages and migration paths in place so existing consumers are guided forward instead of being forced into a same-day rewrite. `v1.8.0` should also state that `CSharpDB.Core` and `CSharpDB.Service` are planned for removal in `v2.0.0`.

### What should release in `v1.8.0`

- `CSharpDB.Primitives` is the primary package named in docs, samples, package tables, and release notes
- `CSharpDB.Core` continues to ship as a compatibility package that forwards users to `CSharpDB.Primitives`
- `CSharpDB.Service` continues to ship, but only as a deprecated compatibility facade over `CSharpDB.Client`
- `CSharpDB.Client` gRPC support ships as a supported feature with clear configuration and upgrade guidance
- docs, package metadata, and obsolete warnings all point to `v2.0.0` as the removal target for compatibility surfaces

### What should not happen in `v1.8.0`

- do not unpublish or remove `CSharpDB.Core`
- do not remove `CSharpDB.Service`
- do not rename public namespaces unless that is separately justified and planned
- do not mix deprecation messaging with silent runtime behavior changes

### Hard vs Soft Breaking Changes

Soft breaking changes acceptable in `v1.8.0`:

- package recommendation changes
- deprecation warnings and obsolete annotations
- stricter validation for gRPC endpoint/options when users explicitly choose `Grpc`
- updated docs and templates that stop steering new consumers toward compatibility packages

Hard breaking changes to defer to a later major release:

- removing `CSharpDB.Core`
- removing `CSharpDB.Service`
- making compatibility-only packages fail at runtime instead of forwarding
- large namespace or API surface renames

### Package Release Shape

Recommended package story for the `v1.8.0` release:

- publish `CSharpDB.Client` with working gRPC transport
- publish `CSharpDB.Primitives` as the canonical low-level package
- continue publishing `CSharpDB.Core` as a compatibility package
- continue publishing `CSharpDB.Service` as a deprecated compatibility package
- update the umbrella `CSharpDB` package so its dependency graph and README reflect the new preferred surfaces

### User-Facing Release Messaging

The `v1.8.0` release notes should have a dedicated `Breaking Changes and Migration` section with three parts:

1. `Package migration`
   `CSharpDB.Core` is replaced by `CSharpDB.Primitives` as the primary package name, while the compatibility package remains available.

2. `Service migration`
   New application code should use `CSharpDB.Client` directly. `CSharpDB.Service` still works in `v1.8.0`, but it is compatibility-only and should not receive new first-class features.

3. `Transport behavior`
   `Grpc` is now implemented, so users choosing that transport get real endpoint validation and real network behavior instead of a placeholder not-implemented path.

### Recommended Timeline

1. Implement compatibility-safe changes first
   Ship `Grpc`, docs, benchmarks, package guidance, and deprecation annotations without removing existing compatibility packages.

2. Publish `v1.8.0` with explicit migration notes
   Treat this as the transition release that announces the preferred package/API story.

3. Hold one release cycle for migration feedback
   Use the next minor release to fix migration pain, docs gaps, or compatibility issues found by users.

4. Remove compatibility surfaces in `v2.0.0`
   If migration coverage is in place, remove `CSharpDB.Core` compatibility packaging and `CSharpDB.Service` in `v2.0.0`, with the final pre-release notes calling out exact upgrade steps.

### Required Release Artifacts

- `RELEASE_NOTES.md` entry with a dedicated breaking-change and migration section
- upgrade examples showing `CSharpDB.Core` -> `CSharpDB.Primitives`
- upgrade examples showing `CSharpDB.Service` -> `CSharpDB.Client`
- obsolete/deprecation messaging in compatibility package docs
- explicit `v2.0.0` removal timeline in compatibility package docs and package descriptions
- benchmark summary so transport changes are explained with measured data

## Risks

- The current `ICSharpDbClient` surface may be broader than what is practical to expose over gRPC in one release
- The daemon contract may need DTO shaping that differs from direct engine types
- Deprecating `CSharpDB.Service` without clear migration examples could create confusion for current ASP.NET Core and Blazor hosts
- Package-rename messaging can become noisy if compatibility package behavior is not explained cleanly
- Benchmark results can be misleading if dataset size, warmup behavior, or machine specs are not controlled and documented

## Release Exit Criteria

`v1.8.0` is ready when all of the following are true:

- a documented gRPC path works end to end from `CSharpDB.Client` to the daemon
- `docs/service-daemon/README.md` reflects the actual shipping transport story
- `CSharpDB.Primitives` is the canonical package name across release-facing docs
- `CSharpDB.Service` is clearly documented as deprecated compatibility-only infrastructure
- upgrade notes exist for both package migration and service-layer migration
- the `v2.0.0` removal plan is documented anywhere compatibility packages are recommended today
- benchmark coverage exists and `v1.8.0` has a published performance baseline

## Summary

`v1.8.0` should be the release that turns the transport abstraction into a real daemon-backed client story, while also cleaning up old package boundaries. The repo should leave this release with one clear application API (`CSharpDB.Client`), one clear remote-host story (`CSharpDB.Daemon`), one clear primitives package (`CSharpDB.Primitives`), and one explicitly temporary compatibility layer (`CSharpDB.Service`).
